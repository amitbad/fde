MLflow & the Databricks ML Platform

FDEs are increasingly asked to stand up the ML lifecycle, not just data pipelines. You don't need to be a data scientist, but you must be able to explain and deploy the platform plumbing: experiment tracking, the Model Registry (now in Unity Catalog), model serving, and the Feature Store. This chapter gives you exactly that operational fluency.

⏱️ ~2.5 hours🧪 1 lab🎯 7 interview questionsTracking FREE, Serving PAID
Open-source MLflow runs anywhere

8.1 The Four MLflow Components

MLflow is an open-source framework (created by Databricks) covering the ML lifecycle. Know the four pillars and what each solves.

Component Solves Key objects
Tracking "Which run/params produced this metric?" Experiments, runs, params, metrics, artifacts.
Models Package a model in a standard "flavor" so anything can serve it. The MLmodel format, flavors (sklearn, pyfunc…).
Model Registry Versioning, aliases, and governance of models. Registered models, versions, aliases/stages.
Serving / Deployment Expose a model as a REST endpoint or batch scorer. Serving endpoints, batch inference.
Tracking works on Community/Free Edition

8.2 Experiment Tracking

Every training run logs its parameters, metrics, and artifacts so results are reproducible and comparable. On Databricks, autologging captures most of this automatically.

Python — explicit tracking + autolog
import mlflow
from sklearn.ensemble import RandomForestClassifier
from sklearn.metrics import f1_score

mlflow.sklearn.autolog()                     # auto-captures params, metrics, model

with mlflow.start_run(run_name="rf_baseline") as run:
    model = RandomForestClassifier(n_estimators=200, max_depth=8)
    model.fit(X_train, y_train)
    preds = model.predict(X_test)
    mlflow.log_metric("f1", f1_score(y_test, preds))   # extra custom metric
    mlflow.log_param("feature_set", "v3")
    print(run.info.run_id)                    # reference this run later

FDE value-add

Set a shared experiment path in a Unity Catalog schema so a customer's whole team logs to one governed, searchable place instead of scattered notebooks — instant reproducibility and auditability.

Models-in-UC needs a cloud workspace

8.3 Model Registry in Unity Catalog

Databricks now registers models as first-class Unity Catalog objects (catalog.schema.model), so the same RBAC, lineage, and audit you learned in Chapter 4 apply to models. Modern practice uses aliases (e.g. @champion, @challenger) instead of the legacy Staging/Production stages.

Register a run's model and set an alias
mlflow.set_registry_uri("databricks-uc")

model_uri = f"runs:/{run_id}/model"
mlflow.register_model(model_uri, "main.ml.churn_model")   # -> version N

from mlflow import MlflowClient
c = MlflowClient()
c.set_registered_model_alias("main.ml.churn_model", "champion", version=3)

# Load whatever is currently champion — deployment code never changes
prod = mlflow.pyfunc.load_model("models:/main.ml.churn_model@champion")

Interview framing: aliases vs stages

"Stages (Staging/Production) were a fixed lifecycle. UC aliases are flexible pointers — I promote by moving @champion to a new version, enabling clean blue/green and A/B (@challenger) without touching serving code."

Serving endpoints are a paid feature

8.4 Deployment — Batch vs Real-Time Serving

  • Batch inference: load the model as a Spark UDF and score a whole table. Cheapest, highest throughput, no always-on cost.
  • Real-time serving: Databricks Model Serving hosts the model behind an autoscaling REST endpoint (scales to zero when idle). Use only when you need low-latency, per-request predictions.
Batch scoring with a registered model as a Spark UDF
import mlflow
predict = mlflow.pyfunc.spark_udf(spark, "models:/main.ml.churn_model@champion")
scored = spark.table("main.ml.customers_features").withColumn("churn_prob", predict("*"))
scored.write.mode("overwrite").saveAsTable("main.ml.churn_scores")

📊 Cost decision

Real-time endpoints cost money whenever provisioned (even scale-to-zero has cold-start/uptime tradeoffs). If the customer only needs nightly scores, batch UDF inference is often 10×+ cheaper. Always ask "what's the latency SLA?" before recommending real-time serving.

Feature Store needs a cloud workspace

8.5 Feature Store (Feature Engineering in UC)

The Feature Store lets teams publish reusable feature tables and guarantees the same feature computation at training and inference (eliminating train/serve skew). Features are Delta tables in Unity Catalog with lineage to the models that consume them.

  • Offline store: Delta tables for training/batch scoring.
  • Online store: a low-latency key-value store synced for real-time lookups during serving.
  • Lineage: UC tracks which model version used which feature table version — critical for audits.
Steps 1–3 FREE; 4–5 need cloud

8.6 Hands-On Lab — Track, Register, Score

  1. Train a simple sklearn model in a notebook with mlflow.sklearn.autolog() and a couple of custom metrics.
  2. Open the Experiments UI; compare two runs with different hyperparameters side by side.
  3. Pick the best run and note its run_id.
  4. (Cloud) Register the model into a UC schema and set the @champion alias.
  5. (Cloud) Batch-score a feature table using the spark_udf loader and write predictions to a Delta table.

8.7 Interview Questions & Model Answers

Walk me through the ML lifecycle on Databricks.
Testing: end-to-end platform fluency.
Feature engineering into Feature Store tables → training with MLflow Tracking (autolog params/metrics/artifacts) → register the best model into the Unity Catalog Model Registry with a version and alias → deploy via batch UDF scoring or a real-time serving endpoint → monitor and promote new versions by moving the champion alias. Everything is governed by UC RBAC and lineage.
Why register models in Unity Catalog vs the legacy workspace registry?
Testing: current best practice.
UC makes models first-class governed objects with the same RBAC, lineage, and audit as tables, and works cross-workspace. It uses aliases instead of fixed stages, enabling flexible promotion and A/B. It's the recommended approach today.
Batch vs real-time serving — how do you choose?
Testing: cost/architecture judgement.
By latency SLA. If predictions are consumed on a schedule (dashboards, nightly campaigns), batch UDF scoring is far cheaper and simpler. Real-time serving is for per-request, low-latency needs (fraud checks, recommendations) and adds endpoint cost and ops.
What problem does the Feature Store solve?
Testing: understanding train/serve skew.
It guarantees identical feature computation at training and inference, prevents duplicated feature logic across teams, and provides lineage from features to models. Offline store for training, online store for low-latency serving lookups.
What does mlflow autolog capture and why is it useful?
Testing: reproducibility.
It automatically logs framework params, metrics, the model artifact, and often input signatures, without manual log calls. It makes every run reproducible and comparable, which is the foundation of trustworthy ML delivery.
How would you implement champion/challenger?
Testing: deployment patterns.
Register both models; set @champion on the current best and @challenger on the candidate. Serving loads by alias, so I can route a fraction of traffic to challenger or shadow-score it, compare metrics, then promote by moving the champion alias — no code change.
What's available on Community Edition for ML?
Testing: tier honesty.
MLflow tracking and local training work for learning. UC Model Registry, serving endpoints, and Feature Store need a full cloud workspace.
