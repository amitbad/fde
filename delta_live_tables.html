Chapter 5 · Pipelines

Delta Live Tables — Declarative Production Pipelines

DLT is how FDEs ship reliable, self-healing ETL. Instead of orchestrating notebooks by hand, you declare the tables you want and the quality rules they must meet; DLT manages dependencies, incremental processing, retries, and observability. This chapter covers the medallion architecture, streaming tables vs materialized views, expectations, and Change Data Capture with APPLY CHANGES (SCD 1 & 2).

⏱️ ~3 hours🧪 1 lab🎯 8 interview questionsNeeds a cloud workspace
Paid workspace required (DLT is not on Community Edition)

5.1 Why DLT Exists — Imperative vs Declarative

A hand-built pipeline is imperative: you write notebooks, wire them together in a job, manage checkpoints, handle failures, and hope the order is right. DLT is declarative: you describe each dataset as a function, and the engine derives the dependency graph, decides what to recompute, enforces quality, and gives you lineage and metrics for free.

Concern Hand-built notebooks + Jobs Delta Live Tables
Dependencies You order tasks manually. Inferred from table references.
Incremental load You manage checkpoints & watermarks. Streaming tables handle it.
Data quality Custom asserts, easy to skip. Declarative EXPECTATIONS.
Failure recovery Manual reruns. Automatic retries + idempotent recompute.
Lineage & metrics Build it yourself. Auto UI graph + event log.

5.2 The Medallion Architecture

DLT pipelines are conventionally organized in three layers. Memorize this — it is asked in almost every Databricks interview.

RAW SOURCE (cloud files / Kafka) │ ingest as-is, append-only, minimal transformation ▼ ┌──────────────┐ BRONZE ── raw history, schema-on-read, "source of truth" │ bronze │ └──────┬───────┘ │ clean, dedupe, enforce types, apply expectations ▼ ┌──────────────┐ SILVER ── validated, conformed, joined business entities │ silver │ └──────┬───────┘ │ aggregate, model for consumption ▼ ┌──────────────┐ GOLD ── business-level aggregates / BI / ML features │ gold │ └──────────────┘
  • Bronze: land raw data with as little logic as possible so you can always replay history.
  • Silver: cleanse, deduplicate, enforce schema and quality, join reference data.
  • Gold: curated aggregates and features consumed by dashboards, reports, and ML.

5.3 Streaming Tables vs Materialized Views

DLT has two core dataset types. Choosing correctly is the crux of pipeline design.

  • Streaming table (@dlt.table reading a spark.readStream source): processes each input record exactly once, incrementally. Ideal for append-only ingestion (bronze) where you never want to reprocess old data.
  • Materialized view (batch @dlt.table / CREATE MATERIALIZED VIEW): the result is fully recomputed (or incrementally refreshed by Enzyme) whenever inputs change. Ideal for aggregations and joins (silver/gold) where correctness of the full result matters.
Python DLT — bronze streaming ingest with Auto Loader
import dlt
from pyspark.sql.functions import col, current_timestamp

@dlt.table(
    name="bronze_orders",
    comment="Raw orders ingested incrementally from cloud storage",
    table_properties={"quality": "bronze"}
)
def bronze_orders():
    return (
        spark.readStream
             .format("cloudFiles")                       # Auto Loader
             .option("cloudFiles.format", "json")
             .option("cloudFiles.schemaLocation", "/schemas/orders")
             .load("/mnt/landing/orders/")
             .withColumn("_ingested_at", current_timestamp())
    )
Python DLT — silver, reading the bronze streaming table
@dlt.table(name="silver_orders", comment="Cleaned, typed orders")
def silver_orders():
    return (
        dlt.read_stream("bronze_orders")
           .filter(col("order_id").isNotNull())
           .withColumn("amount", col("amount").cast("decimal(10,2)"))
    )

5.4 Data Quality with Expectations

Expectations are declarative row-level constraints. DLT tracks the pass/fail rate and can drop, fail, or merely warn on violations. This is the feature that gets a customer's data quality "for free".

Decorator On violation Use when
@dlt.expect Keep the row, record the metric (warn). You want visibility but not enforcement.
@dlt.expect_or_drop Drop the bad row, keep the pipeline running. Bad rows are noise you can discard.
@dlt.expect_or_fail Abort the update. A violation means the data is untrustworthy.
Combining expectations on a silver table
@dlt.table(name="silver_orders")
@dlt.expect("valid_timestamp", "order_ts IS NOT NULL")
@dlt.expect_or_drop("positive_amount", "amount > 0")
@dlt.expect_or_fail("known_currency", "currency IN ('USD','EUR','GBP')")
def silver_orders():
    return dlt.read_stream("bronze_orders")

FDE value-add

Expectation metrics land in the pipeline event log (a Delta table). You can build a data-quality dashboard on top of it — showing a customer the % of records dropped per rule over time is a concrete, sellable deliverable.

5.5 Change Data Capture — APPLY CHANGES (SCD 1 & 2)

Source systems emit inserts, updates, and deletes. APPLY CHANGES INTO lets DLT merge a change feed into a target table and maintain Slowly Changing Dimensions without you writing MERGE logic or worrying about out-of-order events.

SQL DLT — SCD Type 1 (overwrite, keep latest only)
CREATE OR REFRESH STREAMING TABLE customers;

APPLY CHANGES INTO live.customers
FROM STREAM(live.bronze_customer_changes)
KEYS (customer_id)
APPLY AS DELETE WHEN operation = 'DELETE'
SEQUENCE BY change_ts          -- resolves out-of-order events
COLUMNS * EXCEPT (operation, change_ts)
STORED AS SCD TYPE 1;
SQL DLT — SCD Type 2 (keep full history with __START_AT/__END_AT)
APPLY CHANGES INTO live.customers_history
FROM STREAM(live.bronze_customer_changes)
KEYS (customer_id)
SEQUENCE BY change_ts
STORED AS SCD TYPE 2;
  • KEYS: the primary key identifying a logical row.
  • SEQUENCE BY: the ordering column so late-arriving events don't overwrite newer state.
  • SCD Type 1: target holds only the current version. Type 2: target keeps history with validity windows.

5.6 Hands-On Lab — A Three-Layer Pipeline

  1. Upload a few JSON files of fake orders to a cloud volume path (e.g. /Volumes/main/landing/orders).
  2. Create a notebook with the three @dlt.table functions above (bronze → silver → gold aggregate by region).
  3. Add the three expectation types to the silver table.
  4. Go to Workflows → Delta Live Tables → Create pipeline, point it at the notebook, choose a target Unity Catalog schema, and set the pipeline to Triggered mode.
  5. Run it. Open the pipeline graph — confirm the auto-inferred bronze→silver→gold DAG and click each node to see row counts and expectation pass rates.
  6. Drop a malformed file into the landing path and rerun; confirm the expect_or_drop rule quarantines the bad rows and the metric appears in the event log.

Triggered vs Continuous

Triggered pipelines process available data then stop (cheapest, batch-like). Continuous pipelines keep clusters running for low-latency streaming (more expensive). Choose Triggered unless the customer has a real sub-minute latency SLA — a frequent cost-optimization win.

5.7 Interview Questions & Model Answers

Explain the medallion architecture and why each layer exists.
Testing: canonical Databricks design knowledge.
Bronze lands raw, append-only data so history can always be replayed. Silver cleanses, types, deduplicates, enforces quality and conforms entities. Gold produces business aggregates and features for BI/ML. Separating them isolates ingestion volatility from business logic and makes reprocessing safe.
Streaming table vs materialized view — when do you use each?
Testing: the core DLT design decision.
Streaming tables process each source record once, incrementally — perfect for append-only ingestion (bronze/silver). Materialized views recompute/refresh the full result when inputs change — perfect for aggregations and joins where the whole answer must be correct (gold). Use streaming for ingest, MVs for aggregates.
How do DLT expectations work and what are the three modes?
Testing: data-quality feature knowledge.
Expectations are declarative row constraints. expect warns and keeps the row, expect_or_drop discards violating rows, expect_or_fail aborts the update. Metrics for every rule are written to the event log so I can build quality dashboards.
A customer has late-arriving, out-of-order CDC events. How do you keep the dimension correct?
Testing: APPLY CHANGES understanding.
Use APPLY CHANGES INTO ... KEYS(pk) SEQUENCE BY event_ts. The SEQUENCE BY column guarantees that older events never overwrite newer state regardless of arrival order. For history I use SCD Type 2; for current-only, Type 1.
Triggered vs continuous pipelines — cost implications?
Testing: cost awareness.
Triggered starts, processes available data, and shuts compute down — cheap and fine for hourly/daily SLAs. Continuous keeps clusters always on for low latency — much more expensive. I default to Triggered and only go Continuous for genuine sub-minute requirements.
How does DLT know what to recompute?
Testing: declarative model understanding.
DLT builds a dependency graph from the table references in your functions. Streaming tables track processed offsets via checkpoints and only handle new data; materialized views are refreshed based on input changes. You never manually order tasks.
Where do DLT observability metrics live?
Testing: operability.
In the pipeline event log, itself a queryable Delta table, plus the pipeline UI graph. It records row counts, expectation pass/fail rates, durations, and failures — the basis for SLAs and quality dashboards.
Can you use DLT on Databricks Community Edition?
Testing: honesty about tiering.
No — DLT requires a full cloud-linked workspace (Premium/Enterprise). On Community Edition I'd demonstrate the same medallion pattern with plain Structured Streaming + Delta MERGE to prove the concepts.
