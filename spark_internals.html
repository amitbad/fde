Spark Internals — How Your Code Actually Runs

You cannot tune what you do not understand. This chapter builds the mental model of the Spark execution engine from first principles: the cluster topology, the logical-to-physical journey through Catalyst, jobs/stages/tasks, partitions, the memory model, shuffle mechanics, and Adaptive Query Execution. Every later performance fix is an application of something here.

2.1 Cluster Topology: Driver, Executors, Slots

A Spark application is a distributed program. Understanding which machine does what is the foundation for reading the Spark UI later.

┌─────────────────────────── Spark Application ───────────────────────────┐ │ │ │ ┌──────────────┐ schedules tasks ┌──────────────────┐ │ │ │ DRIVER │ ───────────────────────────▶ │ EXECUTOR 1 │ │ │ │ (1 JVM) │ │ cores = 4 slots │ │ │ │ - SparkSession │ memory = 16 GB │ │ │ │ - DAGScheduler │ [task][task] │ │ │ │ - TaskScheduler│ ◀────────────────────────── │ [task][task] │ │ │ │ - keeps RDD lineage results / heartbeats └──────────────────┘ │ │ └──────────────┘ ┌──────────────────┐ │ │ │ EXECUTOR 2 │ │ │ │ cores = 4 slots │ │ │ │ [task][task]... │ │ │ └──────────────────┘ │ └──────────────────────────────────────────────────────────────────────────┘
  • Driver: the single JVM that runs your main() / notebook cell. It builds the query plan, splits work into tasks, schedules them, and collects results. If you call .collect() on a huge DataFrame, all that data flows back into this one JVM — the classic Driver OOM.
  • Executor: a JVM on a worker VM that actually processes data. It has a fixed number of cores (task slots) and a fixed memory budget.
  • Slot / core: one task runs per slot at a time. An executor with 4 cores runs 4 tasks in parallel. Total cluster parallelism = executors × cores.

Why this matters for an FDE

If a customer's cluster has 10 executors × 4 cores = 40 slots but their job only produces 8 partitions, 32 slots sit idle and they are paying for hardware doing nothing. Matching partition count to slot count is a recurring, high-value fix.

2.2 Transformations vs Actions (Lazy Evaluation)

Spark does nothing when you chain transformations. It only builds a recipe (a logical plan). Work is triggered when an action is called. This is why a cell with five .filter()/.join() calls returns instantly, but .count() takes minutes.

PySpark — nothing executes until the action
# Transformations — lazy, just build the plan. Returns instantly.
df = spark.read.table("sales.orders")          # narrow
df2 = df.filter(df.amount > 100)               # narrow (no shuffle)
df3 = df2.groupBy("region").sum("amount")      # wide  (shuffle!)

# Action — NOW Spark builds stages and runs tasks.
df3.show()                                     # triggers a Job

Narrow vs Wide dependencies

  • Narrow transformation (filter, select, withColumn): each input partition contributes to exactly one output partition. No data moves between executors. Cheap.
  • Wide transformation (groupBy, join, distinct, orderBy): output partitions depend on many input partitions, so data is shuffled across the network. Expensive — this is where a stage boundary is created.

⚠️ Common gotcha

Because evaluation is lazy, an error in a transformation often only surfaces when the action runs — sometimes many cells later. When debugging a customer notebook, look at the action that failed, then walk the plan backward.

2.3 The Catalyst Optimizer — Logical to Physical

When you trigger an action, your DataFrame code is compiled by Catalyst, Spark's cost-based optimizer, through four stages:

Your DataFrame / SQL │ ▼ 1. UNRESOLVED LOGICAL PLAN ── "what" you asked for, columns not yet verified │ (Analysis: resolve columns/tables against the catalog) ▼ 2. RESOLVED LOGICAL PLAN ── columns & types validated │ (Logical Optimization: rule-based rewrites) ▼ 3. OPTIMIZED LOGICAL PLAN ── predicate pushdown, constant folding, column pruning │ (Physical Planning: generate candidate strategies, pick by cost) ▼ 4. PHYSICAL PLAN ── chosen join algorithm, exchange (shuffle) operators │ (Codegen: compile to JVM bytecode via Whole-Stage CodeGen) ▼ RDDs executed as tasks

Key optimizations to be able to name in an interview

  • Predicate pushdown: filters are pushed as close to the data source as possible, so Parquet/Delta only reads the rows/files that can match.
  • Column pruning: only the columns you reference are read off disk (columnar formats make this cheap).
  • Constant folding: WHERE 1=1 AND year=2024 is simplified at compile time.
  • Whole-Stage Code Generation: collapses a chain of operators into a single tight JVM function, removing virtual-call overhead.
Read the plan yourself — the single most useful diagnostic command
df3.explain(mode="formatted")

# Look for these operators in the output:
#   * PhotonScan / FileScan parquet   -> is predicate pushdown happening? (PushedFilters)
#   * Exchange hashpartitioning       -> a SHUFFLE (stage boundary, expensive)
#   * BroadcastHashJoin vs SortMergeJoin  -> which join strategy did Catalyst pick?
#   * AQEShuffleRead coalesced         -> Adaptive Query Execution kicked in

2.4 Jobs → Stages → Tasks (The Spark UI Hierarchy)

This is the vocabulary the Spark UI uses. Master it and the UI becomes readable.

Unit Created by Boundary Rule of thumb
Job One action (count, write, show). The action. 1 action = 1 job.
Stage A shuffle boundary inside a job. Each wide transformation (Exchange). N shuffles = N+1 stages.
Task One unit of work per partition, per stage. One partition. #tasks in a stage = #partitions.

Reading it in practice

A job with a groupBy has 2 stages: Stage 0 reads + pre-aggregates (map side), then an Exchange shuffles, then Stage 1 finishes the aggregation (reduce side). If Stage 1 has 200 tasks and 199 finish in 2s but 1 runs for 4 minutes, you are looking at data skew — the topic of Chapter 3.

2.5 Partitions — The Unit of Parallelism

A partition is a chunk of the dataset processed by exactly one task on one slot. Partition count is the single biggest lever on parallelism and shuffle cost.

  • Too few partitions: idle slots, giant tasks, memory pressure and spill.
  • Too many partitions: scheduling overhead dominates, tiny wasteful tasks ("small files"/"small tasks" problem).
  • Default shuffle partitions: spark.sql.shuffle.partitions defaults to 200. On a small cluster that is often too many; on a huge job too few. AQE now tunes this automatically (see 2.7).
Inspect and control partitioning
print(df.rdd.getNumPartitions())          # how many partitions right now

df_repart = df.repartition(64, "region")  # full shuffle to 64 parts, hashed by region
df_coal   = df.coalesce(8)                 # merge to 8 parts WITHOUT full shuffle (narrow)

# Set shuffle partitions for a session
spark.conf.set("spark.sql.shuffle.partitions", 64)

⚠️ repartition vs coalesce

repartition(n) triggers a full shuffle and can both increase or decrease partitions and rebalance skew. coalesce(n) only reduces partitions and avoids a shuffle by merging adjacent ones — cheaper, but can create uneven partitions. Use coalesce before writing fewer output files; use repartition to fix skew.

2.6 The Executor Memory Model & Spill

Each executor's JVM heap is divided into regions. Knowing them explains why jobs spill or OOM.

Executor JVM heap (spark.executor.memory) ┌───────────────────────────────────────────────────────────┐ │ Reserved (300 MB, fixed) │ ├───────────────────────────────────────────────────────────┤ │ Unified Memory (spark.memory.fraction, default 0.6) │ │ ┌───────────────────────────┬───────────────────────────┐ │ │ │ Storage (cached data) │ Execution (shuffles, │ │ │ │ can be evicted │ joins, sorts, aggs) │ │ │ │ <───── boundary floats ─────────────────────────────> │ │ │ └───────────────────────────┴───────────────────────────┘ │ ├───────────────────────────────────────────────────────────┤ │ User Memory (0.4) — your UDF objects, data structures │ └───────────────────────────────────────────────────────────┘ + Off-heap / overhead (spark.executor.memoryOverhead) for Python workers, network buffers, Photon native memory.
  • Execution memory holds intermediate shuffle/sort/aggregation buffers. When a task needs more than is free, Spark spills the excess to local disk — correct, but 10–100× slower.
  • Storage memory holds cache()/persist() data. Under pressure it is evicted before execution memory.
  • Overhead / off-heap matters heavily for PySpark UDFs (Python worker processes live outside the heap) and for Photon (native C++ memory). PySpark OOMs are frequently overhead, not heap.

Spill is the #1 silent performance killer

In the Spark UI's Stage detail, the "Spill (Memory)" and "Spill (Disk)" columns tell you a task overflowed. Fixes: more/again-balanced partitions, larger memory-optimized VMs, avoiding giant explode/cross joins, or enabling AQE. Chapter 3 walks the full diagnosis.

Enabled by default on Databricks Runtime

2.7 Adaptive Query Execution (AQE)

AQE re-optimizes the physical plan at runtime using real statistics collected after each shuffle, rather than trusting compile-time estimates. It is on by default in modern Databricks Runtime and is a frequent interview topic. Three things it does:

  • Dynamically coalescing shuffle partitions: if the 200 default shuffle partitions each end up tiny, AQE merges them into fewer, right-sized partitions — no manual spark.sql.shuffle.partitions tuning needed.
  • Dynamically switching join strategy: if a shuffle reveals one side is actually small, AQE converts a SortMergeJoin into a BroadcastHashJoin on the fly.
  • Dynamically handling skew: AQE detects a skewed partition (much larger than the median) and splits it into sub-partitions so it no longer bottlenecks one task.
AQE configuration (defaults shown)
spark.conf.set("spark.sql.adaptive.enabled", True)
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", True)
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", True)

# In df.explain() output, AQE nodes appear as:
#   AQEShuffleRead coalesced      (partitions merged)
#   AQEShuffleRead skewed          (skewed partition split)

Interview framing

"Before AQE I would hand-tune shuffle.partitions and add skew hints. AQE handles the common cases automatically, but it is reactive — it needs a shuffle to gather stats. For extreme, known skew I still combine it with salting or broadcast hints." That answer shows you know both the tool and its limits.

2.8 Hands-On Lab — Watch the Plan Change

Objective: observe stages, shuffles, and AQE by generating data and reading the Spark UI + explain().

  1. Create a notebook. Generate a DataFrame: df = spark.range(0, 20_000_000).withColumn("k", (col("id") % 100)).
  2. Run agg = df.groupBy("k").count() then agg.explain(mode="formatted"). Find the Exchange hashpartitioning node — that is your shuffle / stage boundary.
  3. Call the action agg.count(). Open the Spark UI → Jobs. Confirm 1 job, 2 stages. Click the shuffle stage and note the number of tasks (partitions).
  4. Force skew: skewed = df.withColumn("k", when(col("id") < 19_000_000, lit(0)).otherwise(col("id") % 100)). Group and count again.
  5. In the stage detail, compare task Duration and Shuffle Read Size across tasks — one task should dominate. That is skew.
  6. Toggle spark.conf.set("spark.sql.adaptive.skewJoin.enabled", True) and rerun; look for AQEShuffleRead skewed in the plan and more even tasks.

Expected outcome

You will see the skewed run has one long-pole task while the others idle, and enabling AQE skew handling splits that partition so task durations converge. This is exactly the diagnosis you will perform on real customer jobs in Chapter 3.

2.9 Interview Questions & Model Answers

Explain what happens from the moment you call an action to tasks running.
What they're testing: whether you understand the whole execution pipeline, not just API syntax.
The action triggers a Job. Catalyst compiles my DataFrame through unresolved → resolved → optimized logical plans, then chooses a physical plan (join strategies, exchanges) by cost, and generates JVM bytecode via whole-stage codegen. The DAGScheduler splits the job into stages at shuffle boundaries, and each stage into tasks — one per partition. The TaskScheduler ships tasks to executor slots; executors process their partition, and for wide transformations write shuffle files that the next stage reads.
What is the difference between a narrow and a wide transformation, and why do you care?
What they're testing: understanding of shuffle cost.
Narrow (filter, select) means each input partition maps to one output partition, so no data crosses the network. Wide (groupBy, join, distinct) requires data with the same key to land on the same partition, forcing a shuffle — network + disk I/O and a new stage. I care because shuffles dominate runtime and cost, so I minimize and optimize them (broadcast small tables, pre-aggregate, partition sensibly).
A job is spilling to disk. Walk me through your diagnosis.
What they're testing: practical Spark UI skill.
In the Spark UI stage detail I check the Spill (Memory)/Spill (Disk) columns and per-task shuffle read size. Spill means execution memory was exhausted. I first check for skew (one huge task) — fix with repartition/salting/AQE skew join. If it's uniform pressure, I reduce partition size (more partitions), remove unnecessary wide ops, cache less, or move to memory-optimized VMs. I verify by re-running and confirming spill drops to zero.
What does AQE do and what are its limits?
What they're testing: current, real-world Spark knowledge.
AQE re-optimizes at runtime using post-shuffle statistics: it coalesces tiny shuffle partitions, switches SortMergeJoin to BroadcastHashJoin when a side is actually small, and splits skewed partitions. Its limit is that it's reactive — it needs a shuffle to gather stats, so the first shuffle still runs sub-optimally, and for extreme known skew I complement it with salting or explicit broadcast hints.
Why can a .collect() crash the driver but a .write() not?
What they're testing: driver vs executor responsibilities.
collect() pulls every partition's data back into the single driver JVM, so a large result exceeds driver heap → OOM. write() is distributed: each executor writes its own partitions directly to storage, and only small metadata returns to the driver. For large results I write to a table or use limit()/take(n) instead of collect.
What is whole-stage code generation?
What they're testing: depth beyond the API.
Catalyst fuses a chain of physical operators within a stage into a single generated JVM function, eliminating per-row virtual function calls and intermediate object creation. It makes CPU-bound stages dramatically faster. Photon is the next step — a native C++ vectorized engine that replaces JVM execution for supported operators.
How do you decide the number of shuffle partitions?
What they're testing: sizing intuition.
Rule of thumb: aim for partitions of roughly 128–256 MB and at least as many partitions as total cluster cores so no slot is idle. The default 200 is often wrong for the cluster size. Today I mostly let AQE coalesce automatically, but I sanity-check by dividing shuffle data size by ~128 MB and comparing to core count.
repartition vs coalesce?
What they're testing: a very common trap.
repartition(n) does a full shuffle, can increase or decrease partition count, and rebalances data evenly (good for fixing skew). coalesce(n) only reduces partitions by merging without a shuffle — cheaper but can leave uneven partitions. I use coalesce before writing to reduce output file count, and repartition when I need even distribution.
What is predicate pushdown and which formats support it?
What they're testing: I/O efficiency awareness.
Predicate pushdown moves filters down to the scan so the reader skips data that can't match. Columnar formats like Parquet and Delta support it via file/row-group statistics (min/max) and, with Delta, data skipping + Z-order/liquid clustering. It's why filtering on a clustered column reads far fewer files. Row formats like CSV get little benefit.
Where does PySpark UDF memory live and why does that matter?
What they're testing: Python-on-Spark reality.
Python UDFs run in separate Python worker processes outside the JVM heap, in the executor's memory overhead region, and require serializing rows between the JVM and Python. That causes overhead OOMs and slowness. I prefer built-in Spark SQL functions or, if I must use Python, Pandas/Arrow-optimized UDFs which vectorize the transfer.
Cluster has 40 cores but the stage runs 8 tasks. What's wrong and how do you fix it?
What they're testing: connecting partitions to cost.
Only 8 partitions means only 8 of 40 slots work — 80% of the paid hardware is idle and each task is oversized (spill risk). I'd repartition to at least 40 (ideally a small multiple like 80–120 for scheduling flexibility) or, at the source, split the input files. If it's post-shuffle, ensure AQE isn't over-coalescing and raise shuffle partitions.

FDE Triage Order

Read the Spark UI in this order:

  • 1️⃣ Find the slowest stage first, not the noisiest job summary
  • 2️⃣ Check task duration spread — one long tail usually means skew
  • 3️⃣ Check shuffle read/write and spill before changing configs
  • 4️⃣ Compare tasks versus available slots to spot under-parallelism
  • 5️⃣ Only then decide whether the fix is data layout, join strategy, or cluster shape
