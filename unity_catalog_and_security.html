Unity Catalog & Enterprise Security

Implement bulletproof data governance models using 3-tier namespaces, dynamic PII masking, secure storage credentials, and audit tables.

Paid Cloud Subscription Required

1. Unity Catalog 3-Tier Governance Hierarchy

**Unity Catalog (UC)** is Databricks' flagship data governance model. It replaces legacy DBFS mount structures with a secure, centralized metadata directory. It provides a standard, uniform 3-Tier Namespace hierarchy to reference tables across all cloud clouds: 

 🏢 Metastore (Central core, maps to one region)
  └── 📦 Catalog (The broad department scope, e.g., "finance")
       └── 📂 Schema (The functional dataset, e.g., "audits")
            └── 📄 Table / View / Volume (The actual data resource)

Security Object Privileges

Permissions cascade downwards through the hierarchy using declarative SQL or Python commands. If you grant read privileges on a catalog, the user automatically inherits read privileges on all underlying schemas and tables.

💡 Architecture Decision Pattern

Always configure one Metastore per geographical cloud region. Map multiple workspaces (e.g., workspace_dev and workspace_prod) to the *same* metastore to enable seamless cross-workspace queries and centralized access reporting.

2. Securing the Underlying Storage

In a production-scale Lakehouse, developers should never access cloud storage (AWS S3, Azure Data Lake Gen2) using raw API keys or connection strings hardcoded in notebook cells. FDEs secure the physical storage layer using UC's security abstraction:

The Storage Delegation Pattern

  • Storage Credential: Links an AWS IAM Role or Azure Managed Identity directly to the Metastore. No human user ever sees or manages these cloud keys.
  • External Location: A physical storage path (e.g., s3://my-enterprise-bucket/finance_data/) authorized using a Storage Credential.
  • External Tables: Tables declared inside Unity Catalog that reference files stored securely within an External Location.

FDE Value-Add: Eliminating the "DBFS Mount" Vulnerability

Legacy DBFS mounts make file keys globally accessible to anyone on the cluster. By migrating clients to **Unity Catalog External Locations**, access is strictly checked by the Metastore in real-time, completely passing security compliance audits (GDPR, SOC2).

3. Dynamic PII Masking & Row Filtering

FDEs often implement fine-grained access control policies where sensitive PII data (social security numbers, payroll budgets) is masked dynamically depending on who runs the query.

Dynamic Column Masking

A custom SQL user function determines if the current querying user is an authorized member of a privilege group (e.g., hr_admin). If not, they see a hashed or masked string instead of the raw data. 

-- 1. Create the masking function
CREATE OR REPLACE FUNCTION pii_ssn_mask(ssn STRING)
  RETURN CASE WHEN IS_MEMBER('hr_admin') THEN ssn ELSE '###-XX-XXXX' END;

-- 2. Apply the mask to a column
ALTER TABLE raw_data.hr.employees ALTER COLUMN ssn SET MASK pii_ssn_mask;

Dynamic Row Filtering

Restricts which rows are returned based on the user's regional profile, ensuring regional sales teams can only query datasets relevant to their specific market. 

-- Filter rows so users only see data matching their regional role
CREATE OR REPLACE FUNCTION regional_sales_filter(region STRING)
  RETURN IS_MEMBER('global_admin') OR IS_MEMBER(CONCAT('sales_', region));

-- Apply the filter to the target table
ALTER TABLE main.sales.orders SET ROW FILTER regional_sales_filter ON (order_region);

4. Delta Sharing & Access Auditing

Enterprise clients need to securely share large data tables with external partners without spending money on expensive ETL replication pipelines.

Delta Sharing Protocol

An open protocol developed by Databricks that allows real-time secure sharing of massive datasets.
• No Data Replication: The recipient queries data directly from your storage.
• Open Standard: Recipients can query shared tables using standard PowerBI, pandas, or external Spark without even owning a Databricks subscription.

System Access Auditing

All user actions, cluster creations, and SQL query records are automatically cataloged in UC's system schemas. FDEs query these logs directly to monitor access compliance: 

-- Query the system table to audit sensitive table queries
SELECT user_identity.email, action_name, request_metadata.table_name, timestamp
FROM system.access.audit
WHERE request_metadata.table_name = 'raw_data.hr.employees'
ORDER BY timestamp DESC;

Interview Questions & Model Answers

Explain the Unity Catalog object hierarchy.
Testing: the core governance model.
A regional metastore contains catalogs, which contain schemas, which contain tables/views/volumes/models — the 3-level namespace catalog.schema.object. Privileges cascade down, so granting on a catalog inherits to its schemas and tables. One metastore per region, with multiple workspaces attached for centralized, cross-workspace governance.
How do storage credentials and external locations remove the "hardcoded key" problem?
Testing: secure storage delegation.
A storage credential wraps a cloud IAM role/managed identity; an external location authorizes a specific storage path using that credential. Users are granted on the UC object, never on raw keys, and access is checked by the metastore at query time — eliminating DBFS mounts and shared keys that fail audits.
How would you restrict PII so only HR sees SSNs, dynamically?
Testing: fine-grained access control.
A column mask function returning the real value only when IS_MEMBER('hr_admin'), else a masked string, applied via ALTER TABLE ... SET MASK. Combined with a row filter for regional restriction. The same physical table serves everyone, masked per identity — no data duplication.
What is Delta Sharing and why is it powerful?
Testing: data collaboration.
An open protocol for secure, real-time sharing of large datasets without replication — recipients query directly from the provider's storage, and can even be non-Databricks (pandas, PowerBI, Spark). It avoids costly ETL copies and keeps a single governed source of truth.
How do you prove who queried a sensitive table?
Testing: auditability.
Query system.access.audit, which logs user identity, action, object, and timestamp for every operation. I build a compliance dashboard on it, satisfying SOC2/GDPR who-accessed-what requirements.
What does UC data lineage give you and why does it matter to an FDE?
Testing: operability + trust.
Automatic table- and column-level lineage across queries, notebooks, and pipelines. It's invaluable for impact analysis ("what breaks if I change this column?"), audits, and debugging — concrete value I can show a customer immediately.
