Databricks' 2025–2026 strategic pivot: every data platform is becoming an AI platform. FDEs must wire up Vector Search, Foundation Model APIs, Genie, and Lakehouse Monitoring — not just pipelines.

11.1 Mosaic AI — The Umbrella

Mosaic AI is Databricks' unified AI layer that sits on top of the Lakehouse. It is not a single product — it is the brand for a set of capabilities that covers the full AI lifecycle from raw data to a deployed, governed, production AI application.

Capability What it does FDE touch-point
Foundation Model APIs Pay-per-token access to curated LLMs (DBRX, Llama, Mistral) + external model routing (OpenAI, Anthropic, Azure OAI) Wire customer apps to models via a single SDK endpoint
Vector Search Managed vector database synced to Delta tables — enables RAG without external infra Build retrieval-augmented generation pipelines on existing lakehouse data
AI/BI (Genie) Natural-language BI: ask questions about data in plain English, get charts and SQL Stand up a Genie Space in a customer BI workflow in hours
Lakehouse Monitoring Statistical data quality and drift monitoring on Delta tables, produces analysis tables in UC Replace ad-hoc DQ scripts with governed, scheduled monitor profiles
Databricks Apps Host Streamlit/Gradio/Dash web apps natively inside Databricks with full UC auth Deliver customer-facing AI demos/tools without external cloud infra
Model Serving (expanded) Single endpoint serves custom models, Foundation Models, and external models behind one API A/B routing, traffic splitting, pay-per-token vs provisioned throughput

The FDE mental model

Think of Mosaic AI as the intelligence layer on top of the Lakehouse: Delta + Unity Catalog stores data; Mosaic AI adds embeddings, retrieval, model serving, and NL interfaces on top of that governed data — without copying it out.

Requires pay-per-token or provisioned throughput workspace

11.2 Foundation Model APIs

Foundation Model APIs (FMAPI) let you call curated open-source models (DBRX Instruct, Llama 3, Mistral, BGE-Large-EN for embeddings) and route to external providers (OpenAI, Anthropic, Azure OAI) — all through a single, governance-wrapped SDK endpoint tracked in Unity Catalog.

Two modes of access

  • Pay-per-token: call Databricks-hosted models without provisioning clusters. Bill appears in system.billing.usage like any DBU workload. Best for dev/test or bursty inference.
  • Provisioned Throughput: dedicated model capacity (tokens/second guaranteed). Use when a customer needs consistent low-latency SLA for production traffic.
Python — call DBRX Instruct via the OpenAI-compatible SDK
from openai import OpenAI

client = OpenAI(
    api_key="dapi...",                               # Databricks personal access token
    base_url="https://<workspace>.azuredatabricks.net/serving-endpoints"
)

response = client.chat.completions.create(
    model="databricks-dbrx-instruct",
    messages=[
        {"role": "system", "content": "You are a SQL expert."},
        {"role": "user",   "content": "Rewrite this query to use window functions."}
    ],
    max_tokens=512
)
print(response.choices[0].message.content)

External Model Routing

An External Models endpoint proxies calls to OpenAI, Anthropic, or Azure OAI via a UC-governed secret — so the customer's access key is never in notebook code and all calls are audited.

Create an external model endpoint (Terraform / REST)
# via Databricks REST API
POST /api/2.0/serving-endpoints
{
  "name": "openai-gpt4o-proxy",
  "config": {
    "served_entities": [{
      "name": "gpt4o",
      "external_model": {
        "name": "gpt-4o",
        "provider": "openai",
        "task": "llm/v1/chat",
        "openai_config": { "openai_api_key": "{{secrets/openai/key}}" }
      }
    }]
  }
}

FDE cost framing

Pay-per-token = no idle cost, great for <1K calls/day. Provisioned throughput = flat cost regardless of usage, break-even at ~500K tokens/day. Always model the volume before recommending PT to a customer.

Vector indexes require a cloud workspace

11.3 Vector Search — Managed Embeddings for RAG

Databricks Vector Search is a serverless vector database that syncs automatically from a Delta table. It eliminates the need to manage Pinecone, Weaviate, or FAISS externally — the index lives in your Lakehouse and is governed by Unity Catalog.

RAG architecture on Databricks

Source documents (PDFs, wikis, code) │ ▼ (ETL pipeline — Delta Live Tables or notebook) Delta Table ←──── Unity Catalog governed │ ▼ (Vector Search sync — incremental, scheduled) Vector Search Index ←──── auto-updated when source changes │ ▼ (at query time) Retrieval: top-K chunks by cosine similarity │ ▼ Prompt = user_question + retrieved_chunks │ ▼ Foundation Model API (DBRX / GPT-4o / Llama) │ ▼ Answer
Python — create a Vector Search index and query it
from databricks.vector_search.client import VectorSearchClient

vsc = VectorSearchClient()

# 1. Create a Vector Search endpoint (once per workspace)
vsc.create_endpoint(name="fde_vs_endpoint", endpoint_type="STANDARD")

# 2. Create a Delta Sync index — keeps itself updated as source Delta table changes
vsc.create_delta_sync_index(
    endpoint_name="fde_vs_endpoint",
    source_table_name="main.docs.knowledge_base",   # UC table
    index_name="main.docs.knowledge_base_vs_idx",
    pipeline_type="TRIGGERED",                       # or CONTINUOUS
    primary_key="chunk_id",
    embedding_source_column="content",
    embedding_model_endpoint_name="databricks-bge-large-en"
)

# 3. Query — returns top-K similar chunks
idx = vsc.get_index("fde_vs_endpoint", "main.docs.knowledge_base_vs_idx")
results = idx.similarity_search(
    columns=["chunk_id", "content", "source_url"],
    query_text="What is Liquid Clustering?",
    num_results=5
)

⚠️ FDE gotcha — chunking strategy matters

Poor chunk size (too large = noisy retrieval, too small = missing context) degrades RAG quality more than model choice. For enterprise docs: 512–1024 tokens with 15% overlap is a safe starting point. Expose chunk-size as a pipeline parameter so the customer can tune it.

TRIGGERED vs CONTINUOUS sync

  • TRIGGERED: runs a pipeline sync on demand or scheduled — cheaper, ~minutes of latency. Use for knowledge bases that update daily.
  • CONTINUOUS: near-real-time sync as rows change — higher cost. Use only when retrieval must reflect sub-minute source changes (e.g., live support tickets).
Genie requires Unity Catalog + SQL Warehouse

11.4 AI/BI Genie — Natural Language BI

Genie is Databricks' conversational BI layer. Users ask questions in plain English ("Show me top 10 customers by revenue last quarter") and Genie generates SQL, runs it against UC-governed tables, and returns a chart or table — no SQL knowledge required.

Anatomy of a Genie Space

  • Tables: point Genie at specific UC tables (not the whole catalog — be selective).
  • Instructions: natural-language context about the business domain, date conventions, KPI definitions — the more you provide, the more accurate Genie's SQL.
  • SQL Functions / Verified Questions: pre-vetted SQL snippets Genie can call for complex metrics, preventing hallucinated aggregations.
  • Access control: inherits Unity Catalog RBAC — users only see columns they're permitted on.
Sample Genie Instructions (paste in Genie Space UI)
## Business Context
This is an e-commerce sales database. All revenue figures are in USD.
"Active customer" means a customer with a purchase in the last 90 days.

## Date Conventions
- fiscal_year starts Feb 1. When user says "last quarter" use fiscal_quarter.
- Use 'order_date' for time filtering, not 'created_at'.

## Key Metrics
- GMV = SUM(order_total) — always exclude cancelled orders (status != 'CANCELLED')
- Net Revenue = GMV - SUM(refund_amount)
- Repeat rate = COUNT(DISTINCT customer_id with >1 order) / COUNT(DISTINCT customer_id)

FDE delivery pattern

Stand up a Genie Space in the first week of a customer engagement as a quick win — business users get self-serve analytics immediately while you build the proper gold-layer tables underneath. This builds stakeholder trust faster than any presentation.

Lakehouse Monitoring is a paid feature

11.5 Lakehouse Monitoring — Data Quality & Drift at Scale

Lakehouse Monitoring attaches a statistical monitor to any Delta table in UC. It automatically produces two analysis tables — profile_metrics and drift_metrics — that you can query, dashboard, or alert on. No custom DQ pipeline to maintain.

Monitor types

  • TimeSeries monitor: detects data drift and anomalies sliced by a timestamp column. Best for time-partitioned fact tables and streaming sinks.
  • InferenceLog monitor: for ML model scoring tables — tracks prediction drift, label drift, and feature drift between model runs.
  • Snapshot monitor: profiles a table at each refresh; no time slicing. Good for slowly-changing reference/dimension tables.
Python — attach a TimeSeries monitor
from databricks.sdk import WorkspaceClient
from databricks.sdk.service.catalog import MonitorTimeSeries

w = WorkspaceClient()

w.quality_monitors.create(
    table_name="main.sales.orders",
    assets_dir="/Shared/monitors/orders",
    output_schema_name="main.monitoring",
    time_series=MonitorTimeSeries(
        timestamp_col="order_date",
        granularities=["1 day", "1 week"]
    ),
    slicing_exprs=["region", "channel"],       # drill-down dimensions
    schedule=CronSchedule(quartz_cron_expression="0 0 8 * * ?")  # daily 8am
)

# Query generated drift table
spark.sql("""
    SELECT window.start, column_name, drift_type, drift_value
    FROM main.monitoring.orders_drift_metrics
    WHERE drift_type = 'JS_DIVERGENCE' AND drift_value > 0.1
    ORDER BY window.start DESC
""")

What FDEs use this for

Replace ad-hoc Great Expectations scripts with a governed, automatic monitor. The produced analysis tables sit in UC with lineage — any downstream dashboard querying them is auditable. Pair with a Databricks SQL alert to page on-call when drift exceeds a threshold.

11.6 Databricks Apps — Host AI Tools Natively

Databricks Apps lets you deploy Streamlit, Gradio, Dash, or any Python web framework directly inside a Databricks workspace — no external cloud infra, no networking config. The app inherits the user's UC identity and can query Databricks SQL Warehouses directly.

Why it matters for FDEs

  • Deliver a working RAG chatbot or Genie-style tool to a customer without provisioning Azure App Service / AWS Lambda / GCP Cloud Run.
  • App scales to zero when idle — no always-on server cost.
  • Access to secrets via Databricks secret scope, UC table reads via SQL connector — no credentials in app code.
  • CI/CD via DABs: include an app definition in your DABs bundle to deploy alongside jobs and pipelines.
app.py — minimal Streamlit RAG app on Databricks Apps
import streamlit as st
from databricks import sql
from openai import OpenAI

st.title("Lakehouse Knowledge Assistant")

question = st.text_input("Ask a question about your data:")

if question:
    # 1. Retrieve relevant chunks from Vector Search
    from databricks.vector_search.client import VectorSearchClient
    vsc = VectorSearchClient(disable_notice=True)
    idx = vsc.get_index("fde_vs_endpoint", "main.docs.kb_vs_idx")
    hits = idx.similarity_search(query_text=question, num_results=4,
                                  columns=["content"])
    context = "\n".join([h["content"] for h in hits["result"]["data_array"]])

    # 2. Call Foundation Model API with retrieved context
    client = OpenAI(api_key=st.secrets["DATABRICKS_TOKEN"],
                    base_url=st.secrets["DATABRICKS_HOST"] + "/serving-endpoints")
    resp = client.chat.completions.create(
        model="databricks-dbrx-instruct",
        messages=[
            {"role": "system", "content": f"Answer using this context:\n{context}"},
            {"role": "user",   "content": question}
        ]
    )
    st.markdown(resp.choices[0].message.content)
databricks.yml — add app to a DABs bundle
bundle:
  name: fde_rag_bundle

resources:
  apps:
    knowledge_assistant:
      name: "knowledge-assistant"
      description: "RAG chatbot over Lakehouse docs"
      source_code_path: ./app
      config:
        env:
          - name: DATABRICKS_HOST
            value: ${workspace.host}

11.7 Predictive Optimization & Lakehouse Federation

Predictive Optimization

Predictive Optimization automatically runs OPTIMIZE and VACUUM on UC-managed tables based on observed query patterns and file statistics — no cron jobs, no manual scheduling. Enable at the catalog or schema level.

Enable Predictive Optimization
-- Enable on a schema (all tables inherit)
ALTER SCHEMA main.sales
    SET TBLPROPERTIES ('delta.predictiveOptimization.managed' = 'auto');

-- Check optimization history
SELECT * FROM system.storage.predictive_optimization_operations_history
WHERE table_name = 'orders'
ORDER BY operation_end DESC LIMIT 20;

FDE value

Replace the customer's cron-based OPTIMIZE job with Predictive Optimization. Same query performance, zero maintenance, and Databricks charges you only when it runs — operational toil drops to zero for file compaction.

Lakehouse Federation

Lakehouse Federation lets Databricks query external databases (PostgreSQL, MySQL, Redshift, BigQuery, Snowflake, SQL Server) directly via catalog.schema.table without ETL. Data stays in the source system; Unity Catalog controls who can query it.

Set up a federated PostgreSQL connection
-- 1. Create a connection (admin once)
CREATE CONNECTION pg_prod
TYPE POSTGRESQL
OPTIONS (
  host 'db.prod.example.com',
  port '5432',
  database 'app_db',
  user SECRET('prod_secrets', 'pg_user'),
  password SECRET('prod_secrets', 'pg_pass')
);

-- 2. Create a foreign catalog
CREATE FOREIGN CATALOG pg_catalog
USING CONNECTION pg_prod
OPTIONS (database 'app_db');

-- 3. Query as if it were a UC table — no ETL
SELECT * FROM pg_catalog.public.customers LIMIT 100;

When to use Federation vs ETL

Federation for: ad-hoc exploration, reporting on live OLTP data, enrichment joins during migration. ETL into Delta for: performance-critical pipelines, offline ML features, when source SLA/availability is uncertain. FDEs often use both — federate first to prove the data model, then ETL the hot path.

Requires cloud workspace with Mosaic AI enabled

11.8 Hands-On Lab — Build a Minimal RAG App

  1. Create a Delta table in UC with 20+ text documents (use any public wiki text).
  2. Create a Vector Search endpoint and a Delta Sync index on that table using databricks-bge-large-en embeddings.
  3. Trigger a sync and verify num_vectors > 0 in the index status.
  4. Write a Python cell that calls similarity_search() and prints the top-3 chunks for a test question.
  5. Call a Foundation Model API endpoint (DBRX or external GPT-4o) with your retrieved chunks as context and print the answer.
  6. (Bonus) Wrap steps 4–5 in a Streamlit app.py and deploy with Databricks Apps.
  7. (Bonus) Attach a Lakehouse Monitor to the Delta source table and inspect the generated profile_metrics table after the first run.

11.9 Interview Questions & Model Answers

What is Mosaic AI and why should an FDE care about it?
Testing: platform-level awareness, 2025-2026 relevance.
Mosaic AI is Databricks' unified AI layer — Foundation Model APIs, Vector Search, AI/BI Genie, Lakehouse Monitoring, and Model Serving under one governance model. FDEs care because customers now ask "how do we add AI to our data" not just "how do we move data." An FDE who can wire a RAG chatbot onto existing UC tables in a day is far more valuable than one who can only build ETL pipelines.
Design a RAG system for a customer who wants a chatbot over their internal documentation.
Testing: end-to-end architecture synthesis.
Ingest docs into a Delta table (bronze → silver cleaned text via DLT). Create a Vector Search endpoint + Delta Sync index using BGE embeddings — TRIGGERED sync nightly. At query time: embed user question, retrieve top-5 chunks, construct a prompt with context, call DBRX Instruct or an external model via FMAPI. Wrap in a Streamlit app deployed via Databricks Apps, governed by UC (users only retrieve chunks their row-filter allows). Log every retrieval to a Delta table for monitoring. Cost: pay-per-token FMAPI — no always-on serving endpoint needed for <500 users.
Pay-per-token Foundation Model APIs vs Provisioned Throughput — how do you choose?
Testing: cost/architecture judgement.
Pay-per-token has no idle cost and scales instantly — right for dev/test, bursty or low-volume (<~500K tokens/day). Provisioned Throughput guarantees latency SLA with dedicated capacity — right for production serving with consistent high volume or strict p99 requirements. I always model daily token volume first; provisioned throughput break-even is typically above 500K–1M tokens/day.
What is Lakehouse Monitoring and how does it differ from Great Expectations?
Testing: DQ tooling positioning.
Lakehouse Monitoring is a first-class UC feature: attach a monitor to a Delta table, it automatically produces governed profile_metrics and drift_metrics tables on a schedule. No code to maintain, lineage tracked, dashboardable immediately. Great Expectations requires custom pipeline code, an Expectations store, and separate orchestration — more flexible but more ops burden. I recommend LHM for standard statistical DQ on Delta tables and GE when customers need highly custom, assertion-level validation logic.
How does Vector Search handle updates to the source Delta table?
Testing: operational depth.
Delta Sync indexes track the Delta transaction log. In TRIGGERED mode, a sync pipeline runs on schedule or on-demand and upserts/deletes changed vectors. In CONTINUOUS mode, it reacts to each transaction commit. The key point: deletes and updates in the source table propagate to the index — no stale vectors. For a nightly-updated knowledge base, TRIGGERED keeps cost low.
A customer wants to query their PostgreSQL OLTP database from Databricks without ETL. What do you use?
Testing: Lakehouse Federation awareness.
Lakehouse Federation. Create a UC Connection of type POSTGRESQL, then a Foreign Catalog — the table appears as pg_catalog.schema.table in Databricks SQL/notebooks, governed by UC RBAC. No data is copied; Databricks pushes predicates to the source. Appropriate for reporting on live OLTP or migration exploration. For performance-critical or availability-sensitive workloads, I'd follow up with a Structured Streaming or DLT ingestion into Delta.
What is Predictive Optimization and when should you enable it?
Testing: Delta/UC operational best practice.
Predictive Optimization auto-runs OPTIMIZE and VACUUM on UC-managed tables based on query patterns and file stats — eliminating manual maintenance jobs. Enable at schema level for most UC-managed fact tables. Exception: disable on tables with very frequent small appends (streaming sinks mid-microbatch) where you control OPTIMIZE timing precisely for cost reasons.
A customer wants business analysts to query data in natural language without writing SQL. What do you recommend?
Testing: AI/BI Genie positioning.
AI/BI Genie — create a Genie Space, point it at the relevant gold-layer UC tables, write clear business instructions (KPI definitions, date conventions, filter defaults), and add verified SQL functions for complex metrics. Analysts get a chat interface; Genie generates and runs the SQL. Access is governed by UC RBAC, so row filters and column masks apply automatically. Deliver this in day one of an engagement as a quick win.
How do you deploy a Streamlit AI app to customers without external cloud infrastructure?
Testing: Databricks Apps awareness.
Databricks Apps. Package the Streamlit app with a databricks.yml app resource definition, deploy via DABs (databricks bundle deploy). The app runs inside the workspace — no external server, no networking, inherits UC identity for data access, uses Databricks secret scopes for keys. Scales to zero when idle, so no always-on cost. CI/CD drops in naturally alongside existing job/pipeline deployments.
What's available for AI/ML on Community Edition?
Testing: tier honesty.
Community Edition has no Mosaic AI features — no Vector Search, no Foundation Model APIs, no Genie, no Apps. You can practise MLflow tracking and local model training, and study the SDK APIs, but all Mosaic AI features require a full cloud workspace. I tell customers this clearly before scoping a proof of concept.
