Case Studies & FDE Customer Scenarios

Real FDE interviews and customer work are rarely about memorizing one default value. They test whether you can diagnose messy systems, pick tradeoffs, communicate with stakeholders, and turn Databricks features into business outcomes.

⏱️ ~4 hours🧪 8 scenario drills🎯 12 interview questionsConceptual FREE
Interview framing

12.1 The FDE Case Study Framework

Use this pattern for almost every customer scenario. It keeps your answer structured and prevents jumping straight to random configs.

  1. Clarify the business impact: SLA missed, bill increased, audit blocked, POC delayed, analysts waiting, or executives losing trust.
  2. Establish current state: workspace tier, compute type, data size, table layout, job schedule, ownership, security model, and cost baseline.
  3. Inspect evidence: Spark UI, query profile, system tables, audit logs, DLT event logs, cluster metrics, lineage, and failed records.
  4. Find the dominant bottleneck: skew, spill, small files, scan volume, inefficient joins, over-provisioned compute, governance gaps, or unclear semantics.
  5. Recommend the smallest safe fix first: change one or two high-confidence levers, measure before and after, then automate.
  6. Convert the fix into a repeatable pattern: dashboards, policies, DABs, runbooks, alerts, and stakeholder-facing success metrics.

Answer shape in interviews

Start with discovery, then diagnostics, then remediation, then prevention. Interviewers trust candidates who say what they would measure before saying what they would change.

Spark + cost + stakeholder communication

12.2 Case Study — Nightly ETL Runs 6 Hours and Misses SLA

Customer situation

A retail customer has a nightly sales aggregation job scheduled at 1 AM. It used to finish in 90 minutes, now takes 6 hours and blocks dashboards used by regional managers at 8 AM.

Discovery questions

  • What changed recently: data volume, schema, cluster policy, join logic, source system, or table layout?
  • Is the workload CPU-bound, shuffle-bound, IO-bound, or waiting for cloud storage/listing?
  • Are slow stages tied to one large join, one aggregation, or many small file reads?
  • Does the job have an SLA and what business process fails if it misses?

Evidence to inspect

  • Spark UI: stage duration, shuffle read/write, spill, skewed task duration, executor lost events.
  • Query profile: join strategy, scan rows, predicate pushdown, Photon usage, broadcast decisions.
  • Delta history: file counts, OPTIMIZE history, table growth, schema evolution, ZORDER or Liquid Clustering status.
  • Billing/system tables: DBU spend per job run and whether cluster size increased without reducing duration.

Likely root causes

  • Skew: one region, customer, tenant, or null key dominates a join or aggregation.
  • Small files: millions of tiny files make metadata and scan overhead dominate.
  • Bad join strategy: Spark shuffles both large tables when a dimension could be broadcast.
  • Over-partitioning or under-partitioning: default partition counts are not aligned with actual data and cluster shape.

FDE recommendation

  1. Capture a baseline: runtime, DBU cost, slowest stages, shuffle bytes, spill bytes, and output row count.
  2. Enable Photon and AQE if not already enabled and validate compatibility.
  3. If small files dominate, run OPTIMIZE or enable Predictive Optimization for UC-managed tables.
  4. If skew dominates, isolate top skew keys and apply salting, skew hints, or pre-aggregation.
  5. If a small dimension is shuffled, broadcast it explicitly or tune table stats so Spark chooses broadcast.
  6. Publish before/after metrics and add a runbook explaining the chosen fix.

Strong FDE answer

I would not start by setting spark.sql.shuffle.partitions blindly. I would first inspect whether the slow stage is skew, spill, small-file scan, or join strategy. Config changes come after evidence.

Governance + migration

12.3 Case Study — Unity Catalog Migration from Mounts

Customer situation

A financial services customer uses DBFS mounts with shared service principals. Security wants Unity Catalog before a production expansion, but data teams fear breaking hundreds of notebooks.

Key risks

  • Access breakage: notebooks rely on mount paths and workspace-local permissions.
  • Credential sprawl: secrets are spread across notebooks, jobs, and legacy scopes.
  • Ownership confusion: tables exist without clear data owners or grants.
  • Audit gap: shared credentials hide which user accessed sensitive data.

Migration plan

  1. Inventory mounts, external paths, table dependencies, job owners, and notebook references.
  2. Design catalog/schema hierarchy by domain, environment, and sensitivity.
  3. Create storage credentials, external locations, managed locations, and groups through Terraform.
  4. Migrate read paths from /mnt/... to catalog.schema.table or volumes.
  5. Apply row filters and column masks for sensitive fields before production cutover.
  6. Run dual validation: row counts, checksums, lineage, and sample business queries.
  7. Freeze new mount creation and publish a developer migration guide.

Common mistake

Do not make every engineer a metastore admin just to unblock migration. Use domain groups, ownership, grants, and break-glass admin access.

FinOps + operations

12.4 Case Study — Databricks Bill Doubled Overnight

Customer situation

The CFO escalates because Databricks spend doubled in one week. Engineering says workloads are unchanged. The customer wants immediate savings without breaking production.

Investigation path

  • Query system.billing.usage by workspace, SKU, job, cluster, user, and custom tags.
  • Compare current week to previous four weeks by DBU, cloud VM cost, SQL warehouse cost, and serverless usage.
  • Check for all-purpose clusters left running, SQL warehouses without auto-stop, and jobs retried in loops.
  • Correlate expensive jobs with runtime, input data volume, and failed/retried task counts.

Immediate containment

  1. Stop idle all-purpose clusters and enforce autotermination.
  2. Apply cluster policies with max workers, allowed node types, Photon defaults, and mandatory tags.
  3. Set SQL warehouse auto-stop and review serverless warehouse sizing.
  4. Move repeatable production jobs from interactive clusters to Jobs compute.
  5. Create budget alerts and daily spend anomaly dashboards.

Long-term fix

  • Chargeback: map every workload to team, product, environment, and owner.
  • Optimization: tune the top five expensive jobs before touching low-spend workloads.
  • Governance: require tags and restrict expensive compute choices through policies.
  • Education: publish a cost playbook for developers and analysts.
Streaming + reliability

12.5 Case Study — Streaming Job Creates Duplicate Records

Customer situation

A near-real-time ingestion pipeline uses Auto Loader and writes to Delta. Downstream users report duplicate orders after a source outage and replay.

Likely causes

  • Checkpoint path was deleted or changed, causing files to be processed again.
  • Sink write is append-only without idempotent merge logic.
  • Source emits updates or late corrections but pipeline treats all events as new rows.
  • Watermark is too aggressive or missing, causing wrong state handling.

FDE remediation

  1. Stop the pipeline and preserve checkpoint, input files, and target table history.
  2. Identify duplicate keys and replay window using Delta time travel.
  3. Repair target with a deterministic dedupe rule: latest event timestamp, highest version, or source sequence number.
  4. Change the sink to idempotent MERGE or DLT APPLY CHANGES for CDC-style sources.
  5. Lock checkpoint path conventions through DABs and environment-specific config.
  6. Add alerts for duplicate rate, input backlog, and late-arriving event volume.

Interview signal

Exactly-once is an end-to-end design property, not a magic checkbox. Checkpointing helps, but the sink must be idempotent when sources can replay data.

AI + delivery

12.6 Case Study — Two-Week RAG POC for Internal Knowledge

Customer situation

A customer wants a chatbot over policy documents, runbooks, and internal wiki pages. Leadership wants a demo in two weeks, but security requires access controls and auditability.

Architecture

Documents → Bronze Delta → Clean/chunk Silver Table → Vector Search Index │ │ └──────────── Unity Catalog permissions + lineage ──────┘ │ User question → retrieve top-K chunks → Foundation Model API → Databricks App

Delivery plan

  1. Pick one high-value corpus and define success criteria: answer accuracy, citation quality, latency, and adoption.
  2. Ingest documents into Delta with source URL, owner, sensitivity label, and last modified timestamp.
  3. Chunk documents with consistent chunk size and overlap; keep chunk metadata for citations.
  4. Create a Vector Search Delta Sync index in TRIGGERED mode for daily-updated documents.
  5. Use FMAPI pay-per-token for the POC; avoid provisioned throughput until traffic is proven.
  6. Deploy a Streamlit app using Databricks Apps and log every question, retrieved chunk ID, model, latency, and user feedback.
  7. Run an eval set of 50 real questions with expected citations before calling the POC successful.

Security question to ask early

Should users only retrieve chunks they are authorized to read? If yes, design row-level filtering and metadata filtering before the demo, not after.

AI/BI + stakeholder trust

12.7 Case Study — Business Wants Natural Language Analytics

Customer situation

Sales leadership wants to ask questions in plain English instead of waiting for analysts. The data team has gold tables, but KPI definitions are inconsistent across dashboards.

FDE approach

  1. Select a narrow domain: sales pipeline, bookings, renewals, or support tickets.
  2. Point Genie only to curated gold tables, not raw bronze/silver tables.
  3. Write instructions for KPI definitions, fiscal calendar, default filters, and business synonyms.
  4. Add verified questions for the top 20 executive queries.
  5. Validate generated SQL with analysts and mark trusted answers.
  6. Use UC row filters and column masks so executives and regional managers see only permitted data.

Why this is FDE-relevant

A Genie Space can be a week-one credibility win. It shows business value quickly while you continue hardening pipelines, governance, and cost controls underneath.

Migration + architecture tradeoff

12.8 Case Study — Query Operational DB Without ETL

Customer situation

A customer wants Databricks dashboards over PostgreSQL operational data this week. The source team refuses a new ETL pipeline until the data model is proven.

Recommendation

  • Use Lakehouse Federation first for exploration, stakeholder demos, and low-volume reporting.
  • Move to Delta ingestion when workloads become performance-critical, high-concurrency, ML-feature-driven, or source-SLA-sensitive.
  • Keep governance in UC so access can be managed consistently while data remains in the source.

Decision table

Need Federation Ingest to Delta
Fast POC Best fit Slower setup
High concurrency dashboard May overload source Best fit
ML feature engineering Weak fit Best fit
Source data must not move Best fit Not acceptable
Historical replay Limited Best fit
Practice drills

12.9 Scenario Drills

Use these drills out loud. Give yourself five minutes per scenario and answer using discovery, diagnostics, remediation, prevention, and stakeholder communication.

A production job fails only on month-end when data volume spikes 5x. What do you do?
Testing: reliability and scale thinking.
Check whether failure is memory spill, executor loss, source throttling, shuffle explosion, or timeout. Compare month-end run metrics to normal days. Fix the dominant bottleneck, then add backfill-safe retry strategy, autoscaling boundaries, alerts, and a load-test using month-end-like data before the next cycle.
A customer asks you to make every pipeline serverless. How do you respond?
Testing: product judgement, not feature excitement.
I would evaluate workload type, compliance requirements, startup latency, library needs, networking, cost predictability, and operational burden. Serverless can reduce ops and idle cost, but classic compute may be better for strict network controls, custom libraries, or predictable long-running workloads.
A customer wants all raw data accessible to analysts for speed. What risk do you flag?
Testing: governance and customer management.
Raw access can leak PII, create inconsistent metrics, and bypass quality controls. I would propose curated gold views, UC groups, row filters, column masks, and a fast request path for new datasets instead of unrestricted bronze access.
A pipeline has 10,000 small files per day. Is the answer always OPTIMIZE?
Testing: nuance around heuristics.
Not always. First check table size, query pattern, write frequency, latency requirement, and whether Predictive Optimization can manage it. For streaming sinks, compaction timing must avoid fighting active writes. OPTIMIZE is useful when read overhead is material and file compaction cost is justified.
A VP wants a RAG chatbot, but source documents have no ownership or ACLs. What do you say?
Testing: AI governance.
I would not expose the chatbot broadly until document ownership, classification, and access rules are clear. Start with a restricted corpus, log retrievals, include citations, and ensure the app only retrieves chunks the user is authorized to read.

12.10 Interview Questions & Model Answers

How do you explain the difference between a Databricks developer and an FDE?
Testing: role clarity.
A developer can build notebooks or pipelines. An FDE must diagnose ambiguous customer problems, design production architecture, manage cost and governance, communicate tradeoffs to executives, and leave behind repeatable systems like IaC, runbooks, dashboards, and operating models.
When should you mention Spark defaults in a case interview?
Testing: depth without trivia.
Mention defaults as heuristics after you have identified the bottleneck. For example, spark.sql.shuffle.partitions defaulting to 200 is useful context, but the real answer is to inspect shuffle size, task count, core count, skew, and AQE behavior before changing it.
A customer asks for the fastest possible fix. How do you avoid creating long-term debt?
Testing: pragmatic delivery.
I separate containment from remediation. First stabilize the SLA with a low-risk fix. Then schedule hardening: tests, DABs, ownership, monitoring, cost controls, and documentation. I explicitly tell stakeholders which part is a temporary mitigation and which part is the durable solution.
How do you prove your optimization worked?
Testing: measurement discipline.
Define baseline metrics before the change: runtime, DBU cost, cloud cost, rows processed, shuffle bytes, spill, failed tasks, SLA miss rate, and business availability. After the change, compare the same metrics over multiple runs and publish the result in a dashboard or summary.
How do you handle disagreement between security and data engineering teams?
Testing: stakeholder navigation.
I translate both sides into requirements. Security needs auditability, least privilege, and policy enforcement. Engineering needs speed and fewer blockers. Unity Catalog groups, external locations, row filters, masks, and self-service request flows can satisfy both if designed with clear ownership.

FDE Case Answer

Clarify: business impact and current state.

Inspect: Spark UI, query profile, system tables, logs, lineage.

Decide: smallest safe fix based on evidence.

Prove: baseline versus after metrics.

Operationalize: IaC, policies, alerts, dashboards, runbooks.

Do Not Over-Index On

  • Memorizing defaults without knowing when they matter
  • Adding nodes before reading the Spark UI
  • Recommending new features without migration risk analysis
  • Ignoring business SLA, cost, and stakeholder trust
