Chapter 9 · Money

Cost Governance & FinOps

The fastest way to be indispensable at a customer is to cut their bill without cutting capability. This chapter turns the DBU model from Chapter 1 into an operating discipline: tagging, budgets, the system.billing.usage tables, chargeback reporting, and the concrete levers that reduce spend. Money questions decide FDE interviews.

⏱️ ~2 hours🧪 1 lab🎯 8 interview questionsSystem tables need a cloud workspace
Concept applies everywhere

9.1 The Cost Equation, Restated

Every dollar on Databricks is two components. FDEs must be able to recite this on a whiteboard:

TOTAL COST = DATABRICKS COST + CLOUD INFRASTRUCTURE COST (DBUs × $/DBU × hours) (VM + storage + network, billed by AWS/Azure/GCP) where $/DBU depends on: • workload type (Jobs < DLT < All-Purpose < SQL, roughly) • tier (Standard / Premium / Enterprise) • compute type (Classic vs Serverless)
  • Two bills: customers often forget the cloud VM bill is separate from the Databricks DBU bill. Optimizing one without the other is half a solution.
  • Idle is the enemy: the biggest waste is compute that is provisioned but not doing useful work (interactive clusters left on, oversized always-on warehouses).
Mix of FREE-concept and PAID-feature levers

9.2 The Cost-Reduction Levers (Memorize These)

Lever Typical saving How
Move interactive → Jobs compute Large Jobs DBUs are far cheaper than All-Purpose; automate scheduled work.
Autotermination Large Set 10–20 min idle shutdown on every cluster/policy.
Autoscaling min/max Medium Don't pin large fixed clusters; scale with load.
Spot / preemptible workers 50–70% on VMs Use spot for fault-tolerant workers, on-demand driver.
Photon Varies (net win) Higher DBU/hr but shorter runtime → lower total on CPU-heavy SQL/ETL.
Right-size + fix skew/spill Large Chapter 3 fixes cut runtime and node count together.
Serverless SQL auto-stop Medium Warehouses scale to zero between queries.
Trigger.AvailableNow streaming Medium Batch economics for incremental jobs (Ch 6).
File layout: OPTIMIZE + liquid clustering Medium Fewer files scanned = less compute + faster.

How to present this to a customer

Lead with the two or three biggest levers backed by their own usage data (next section), quantify the monthly saving, and show it doesn't reduce capability. That framing — "same outcomes, lower bill" — is what gets FDEs renewed.

9.3 Tagging Strategy (The Foundation of Chargeback)

You cannot allocate or optimize what you cannot attribute. Enforce custom tags on clusters, warehouses, and jobs (ideally via cluster policies / Terraform so they can't be omitted).

Cluster policy fragment forcing cost tags (JSON)
{
  "custom_tags.team":        { "type": "fixed", "value": "data-platform" },
  "custom_tags.cost_center": { "type": "allowlist", "values": ["cc-100","cc-200"] },
  "custom_tags.env":         { "type": "fixed", "value": "prod" },
  "autotermination_minutes": { "type": "fixed", "value": 15 }
}
  • team / cost_center: enables chargeback and showback.
  • env: separates prod vs dev spend.
  • pipeline / project: ties spend to a deliverable.
System tables require Unity Catalog

9.4 Querying Spend with System Tables

Unity Catalog exposes billing as queryable Delta tables under the system catalog. This is the FDE's cost microscope — join usage to pricing and to your tags.

SQL — top 10 spending workloads last 30 days
SELECT
    u.usage_metadata.job_id,
    u.custom_tags.team          AS team,
    u.sku_name,
    SUM(u.usage_quantity)       AS dbus,
    SUM(u.usage_quantity * p.pricing.default) AS est_usd
FROM system.billing.usage u
JOIN system.billing.list_prices p
  ON u.sku_name = p.sku_name
 AND u.usage_unit = p.usage_unit
WHERE u.usage_date >= current_date() - INTERVAL 30 DAYS
GROUP BY 1,2,3
ORDER BY est_usd DESC
LIMIT 10;
SQL — daily spend trend by environment tag
SELECT usage_date, custom_tags.env AS env, SUM(usage_quantity) AS dbus
FROM system.billing.usage
WHERE usage_date >= current_date() - INTERVAL 90 DAYS
GROUP BY usage_date, env
ORDER BY usage_date;

Deliverable

Wrap these queries in a Databricks SQL dashboard: top spenders, spend by team/env, and week-over-week trend. That dashboard is a tangible artifact you hand the customer's platform owner.

9.5 Budgets & Guardrails

  • Budgets: configure account-level budget alerts to email owners when spend crosses thresholds.
  • Cluster policies as guardrails: cap max nodes, forbid huge instance types for non-prod, force autotermination, and restrict All-Purpose usage — prevention beats after-the-fact cleanup.
  • SQL warehouse limits: set max scaling and auto-stop so BI usage can't run away.
Cloud workspace required

9.6 Hands-On Lab — Build a Cost Dashboard

  1. Confirm system.billing.usage and system.billing.list_prices are enabled (account admin).
  2. Run the "top 10 spenders" query; identify the single most expensive workload.
  3. Inspect that workload — is it All-Purpose that should be Jobs? Always-on with no autotermination? Skewed (Ch 3)?
  4. Apply one lever (e.g. switch to Jobs compute + 15-min autotermination via a cluster policy) and estimate the monthly saving.
  5. Turn the queries into a Databricks SQL dashboard with a 90-day trend and a by-team breakdown.

9.7 Interview Questions & Model Answers

A customer's Databricks bill doubled last quarter. Walk me through your investigation.
Testing: structured cost triage — a classic FDE scenario.
I'd query system.billing.usage joined to list_prices and group by workload, SKU, team, and date to find what grew. Usual culprits: interactive All-Purpose clusters left running, missing autotermination, an oversized always-on SQL warehouse, or a new skewed/spilling job burning nodes. I quantify each, apply the highest-impact lever (Jobs compute, autotermination, right-sizing, fixing skew), and stand up a dashboard so it's caught early next time.
Explain the two components of Databricks cost.
Testing: fundamental pricing literacy.
The Databricks charge (DBUs consumed × the per-DBU rate for that workload type/tier) plus the cloud infrastructure charge (VMs, storage, network billed directly by the cloud provider). You must optimize both; Serverless folds infra into the DBU rate.
Why is Jobs compute cheaper than All-Purpose, and what's the tradeoff?
Testing: the single biggest lever.
Jobs compute has a much lower per-DBU rate because clusters are ephemeral and single-purpose. The tradeoff is no interactive/shared use — you can't attach a notebook and explore. So scheduled/automated work goes on Jobs compute; interactive development on shared All-Purpose (with autotermination).
How does tagging enable FinOps?
Testing: attribution.
Enforced custom tags (team, cost_center, env, project) let me attribute every DBU in system.billing.usage to an owner, enabling chargeback/showback and pinpointing waste. I enforce them via cluster policies/Terraform so they can't be skipped.
Is Photon always worth it?
Testing: nuance.
Usually for CPU-heavy SQL/ETL: it costs more DBU/hr but finishes faster, lowering total cost. For light or heavily UDF/Python workloads it may not engage, so I benchmark on the actual workload rather than assume.
How do you prevent cost overruns proactively?
Testing: guardrails mindset.
Cluster policies (max nodes, forced autotermination, restricted instance types, forbid All-Purpose for automation), SQL warehouse auto-stop and max scaling, budget alerts, and a monitoring dashboard. Prevention via policy beats cleanup.
Which system tables do you use and what governs access?
Testing: knowing the data source.
system.billing.usage (DBU consumption + tags), system.billing.list_prices (rates), and system.access.audit for who-did-what. They're Unity Catalog objects, so access is governed by UC permissions and must be enabled by an account admin.
Customer wants lower cost but is scared of losing performance. How do you frame it?
Testing: consulting communication.
I frame every lever as "same or better outcomes, lower bill," backed by their own usage data: fixing skew/spill makes jobs both faster and cheaper; autotermination only kills idle time; Jobs compute changes billing, not capability. I quantify savings and de-risk with a staged rollout on non-prod first.
