Workspace Architecture & Pricing Economics

Master Control/Data Plane models, Multi-Workspace structures, secure networking topologies, and core DBU pricing math.

⏱️ ~2.5 hours🧪 1 lab🎯 6 interview questionsCore FREE, networking PAID

1. Control Plane vs. Data Plane

Databricks separates platform operations into two distinct domains: the Control Plane and the Data Plane. This architecture is critical for security, data isolation, and performance optimization.

Control Plane (Databricks Account)

Resides in Databricks' own cloud provider account. Handles administrative operations, cluster management, the web workspace, SQL query history, metadata storage, security access coordination, and job scheduling. It *never* processes or touches customer production datasets directly.

Data Plane (Customer Account or Databricks Serverless)

Where the actual processing of datasets and compute workloads occurs. Databricks splits this into two physical models:

Deployment Model Where Compute Resides Security & VPC Ownership Resource Boot Time
Classic Compute (VPC/VNet) Inside your cloud account (your AWS VPC or Azure VNet). Fully managed by customer network policies and firewalls. 3 to 7 minutes (virtual machine scaling latency).
Serverless Compute Inside a Databricks-managed cloud account, isolated from other customers. Isolated, rapid ephemeral resources managed by Databricks. Under 10 seconds (fully optimized serverless pools).

💡 Architecture Decision Pattern

Choose Classic Compute when corporate security, compliance, or networking requirements mandate that compute resources run within your own AWS account, Azure subscription, or GCP project (for example, inside customer-managed VPCs/VNets).
Choose Serverless Compute for SQL Warehouses, notebooks, and jobs when you want faster startup times, automatic scaling, reduced operational overhead, and Databricks-managed infrastructure, including patching and lifecycle management.

2. Enterprise Multi-Workspace Architecture

Production environments must separate Dev, QA/Stage, and Production workloads. Large enterprises use dedicated workspace configurations to isolate code testing from sensitive business data.

Isolating Workspaces & Catalogs

  • Sandbox/Dev Workspaces: Dedicated environments for engineering, exploration, and notebook writing. Users can deploy local interactive clusters.
  • QA/Staging Workspaces: Where integration, regression testing, and code package testing occur. This mirrors production configurations.
  • Production Workspaces: Strictly locked down. Workloads are automated using workflows or DABs. Standard developers have no write or run permissions directly in this UI.

FDE Value-Add: Platform Lifecycle Control

Manually deploying tables and workspaces across environments introduces human configuration errors and leads to failed security audits. Deploy workspaces and cluster policies programmatically using **Terraform** so environments remain completely uniform.

Paid Cloud Subscription Required

3. Cloud Network Topology & Security

Securing traffic between users, Databricks-managed services, compute resources, and cloud storage (for example S3, ADLS, or GCS) is critical for enterprise security reviews.

Private Link / Private Endpoints

By default, many connections are secured with TLS over cloud-provider public endpoints. FDEs use Private Link / Private Endpoints on supported deployments when a customer requires private network paths instead of public internet exposure:

  • Front-End private access: Restricts workspace UI and API access to approved corporate networks, VPNs, or private DNS paths.
  • Back-End private connectivity: Keeps supported control-plane to compute / storage communication on private cloud networking instead of public routing.

Customer-Managed Keys (CMK)

Customer-Managed Keys let enterprises control the encryption keys used for supported Databricks storage and services through AWS KMS, Azure Key Vault, or Google Cloud KMS. The real value is compliance control, key rotation, and the ability to revoke access if required. It is not an absolute security magic switch; it is a governance and encryption ownership control that must be designed carefully because disabling the key can also make workloads or data inaccessible.

4. DBU Economics & Sizing Formulas

Databricks pricing is driven by DBUs (Databricks Units), but the full cost model depends on the compute type. For Classic Compute, you pay DBUs plus the underlying cloud VM and storage/network charges. For Serverless, Databricks exposes a published serverless price model where infrastructure is typically bundled differently.

📊 Pricing Comparison: All-Purpose vs. Jobs Compute

Exact DBU pricing varies by cloud, region, workspace tier, workload type, and whether the workload is classic or serverless. The durable rule is that All-Purpose Compute is typically priced higher per DBU than Jobs Compute because it supports interactive notebook usage and longer-lived clusters. In customer conversations, explain the formula first and then plug in that customer's actual rate card.

Real-World DBU Sizing Math

Suppose a daily ingestion pipeline runs for 10 hours on a cluster of 4 nodes (plus 1 driver, totaling 5 VMs). Each VM size (e.g., m5.2xlarge) is assumed for illustration to consume 1.5 DBUs per hour. 

Total Cluster DBUs per Hour:
  5 VMs × 1.5 DBUs/hour = 7.5 DBUs/hour

Daily DBUs:
  7.5 DBUs/hour × 10 hours = 75 DBUs/day

General Formula:
  Daily DBU cost = Total daily DBUs × workload-specific DBU rate

Illustrative Example Only:
  If interactive rate = $0.40/DBU, then 75 × $0.40 = $30.00/day in DBU charges
  If jobs rate = $0.15/DBU, then 75 × $0.15 = $11.25/day in DBU charges
  Illustrative DBU savings from moving to Jobs Compute = $18.75/day

For Classic Compute, add cloud VM, storage, and network costs on top of DBUs. For Serverless, the infrastructure charge is usually bundled into the published serverless price, so compare total effective cost rather than DBUs alone. This basic math scales up to meaningful savings when applied across many daily workloads.

5. Lab: Cluster Sizing & DBU Optimization

Objective

Design a cost-estimating spreadsheet for a nightly ingestion workflow and migrate it from interactive cluster execution to an automated job cluster.

Steps to Complete

  • Step 1 (Sizing): Analyze the average data volume of a nightly CSV upload (approx 50GB).
  • Step 2 (Selection): Identify an appropriate node family for the workload. For example, if joins are memory-heavy, choose a memory-optimized VM and record the DBU rate you are assuming for that cluster shape.
  • Step 3 (Configuration): Open the Workflows UI in Databricks. Create a new task.
  • Step 4 (Migration): Link your PySpark notebook to this task. Rather than utilizing an active "All-Purpose" cluster, configure a brand new "Job Cluster" with 1 Driver and 3 Workers, then calculate the estimated DBUs/hour and total daily cost using your own assumptions.
  • Step 5 (Automation): Schedule the task to trigger at 2:00 AM daily.

Interview Questions & Model Answers

Explain the Control Plane vs Data Plane split and why it matters for security.
Testing: the #1 architecture fundamental.
The Control Plane is Databricks-managed (web UI, job scheduling, cluster management, metadata) and never processes customer data. The Data Plane is where compute runs and data is read/written — either Classic (VMs in the customer's own VPC/VNet, so compute and storage access remain in their cloud boundary) or Serverless (Databricks-managed compute). This split lets enterprises keep the data path tightly controlled while Databricks operates the platform.
Classic vs Serverless compute — how do you choose?
Testing: a real architecture decision.
Classic when strict compliance requires VMs inside the customer's own subnets or custom networking/instances. Serverless for fast startup, no idle cost, and lower ops — ideal for SQL warehouses and bursty jobs. Classic adds the cloud VM bill separately; Serverless uses a Databricks-managed price model where infrastructure is bundled differently.
Walk me through the full cost of a 10-hour, 5-node job as interactive vs a job cluster.
Testing: DBU math fluency.
Compute DBUs/hr = nodes × DBU/node. Multiply by hours and the per-DBU rate for the workload type. All-Purpose (interactive) is typically priced higher than Jobs Compute for the same cluster shape, but exact rates depend on cloud, region, tier, and workload. For Classic Compute I would add the cloud VM bill separately; for Serverless I would use the published serverless price model. I'd quantify both options with the customer's actual rate card and then recommend Jobs Compute for scheduled automation if the workflow does not need an always-on interactive cluster.
What is Private Link and which connections does it secure?
Testing: enterprise networking.
Private Link (AWS) / Private Endpoints (Azure) route traffic over the cloud's private backbone instead of the public internet. Front-end private access secures user and API traffic into the workspace; back-end private connectivity secures supported Databricks service traffic to compute and storage. Together with strong identity controls, encryption, and logging, it helps meet stringent enterprise networking requirements.
Why separate Dev/QA/Prod workspaces?
Testing: platform lifecycle.
Isolation of blast radius, data sensitivity, and permissions: developers explore freely in Dev, integration/regression runs in QA, and Prod is locked down to automated deploys only. I provision them identically via Terraform so config drift can't cause an audit failure or a "works in dev" surprise.
What can't you do on Community Edition, and how do you still demonstrate value?
Testing: tier honesty.
You cannot realistically demonstrate enterprise networking features such as customer-managed VPC/VNet deployment, Private Link / Private Endpoints, CMK, or production billing telemetry there. I'd say so plainly, prove the architectural concepts and Spark fundamentals in the free environment, and explain exactly what changes when you move to a paid trial or enterprise workspace.

Trial vs. Enterprise Limits

Databricks Community Edition is free but operates entirely in a shared control plane. You cannot connect cloud virtual networks, provision customer-managed VPCs/VNets, configure Private Link / Private Endpoints, use CMK-backed enterprise controls, or inspect real cloud-cost and billing telemetry.

To practise network security, private connectivity, and CMK in a real environment, you need a paid trial or enterprise workspace connected to an AWS, Azure, or Google Cloud account.
