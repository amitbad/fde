Workspace Architecture & Pricing Economics

Master Control/Data Plane models, Multi-Workspace structures, secure networking topologies, and core DBU pricing math.

Trial / Community Compliant

1. Control Plane vs. Data Plane

Databricks separates platform operations into two distinct domains: the Control Plane and the Data Plane. This architecture is critical for security, data isolation, and performance optimization.

Control Plane (Databricks Account)

Resides in Databricks' own cloud provider account. Handles administrative operations, cluster management, the web workspace, SQL query history, metadata storage, security access coordination, and job scheduling. It *never* processes or touches customer production datasets directly.

Data Plane (Customer Account or Databricks Serverless)

Where the actual processing of datasets and compute workloads occurs. Databricks splits this into two physical models:

Deployment Model Where Compute Resides Security & VPC Ownership Resource Boot Time
Classic Classic (VPC/VNet) Inside Customer's cloud subscription (VPC / VNet). Fully managed by customer network policies and firewalls. 3 to 7 minutes (virtual machine scaling latency).
Serverless Compute Inside a Databricks-owned secure, virtualized compute account. Isolated, rapid ephemeral resources managed by Databricks. Under 10 seconds (fully optimized serverless pools).

💡 Architecture Decision Pattern

Choose Classic Classic when highly strict corporate compliance mandates that compute Virtual Machines *must* exist within customer-owned subnets. Choose Serverless for SQL Warehouses and Serverless Sprints to bypass cluster startup lag, eliminate idle instance costs, and let Databricks manage security patches automatically.

2. Enterprise Multi-Workspace Architecture

Production environments must separate Dev, QA/Stage, and Production workloads. Large enterprises use dedicated workspace configurations to isolate code testing from sensitive business data.

Isolating Workspaces & Catalogs

  • Sandbox/Dev Workspaces: Dedicated environments for engineering, exploration, and notebook writing. Users can deploy local interactive clusters.
  • QA/Staging Workspaces: Where integration, regression testing, and code package testing occur. This mirrors production configurations.
  • Production Workspaces: Strictly locked down. Workloads are automated using workflows or DABs. Standard developers have no write or run permissions directly in this UI.

FDE Value-Add: Platform Lifecycle Control

Manually deploying tables and workspaces across environments introduces human configuration errors and leads to failed security audits. Deploy workspaces and cluster policies programmatically using **Terraform** so environments remain completely uniform.

Paid Cloud Subscription Required

3. Cloud Network Topology & Security

Securing the flow of data between the Control Plane, customer VPC, and cloud storage (e.g., S3/ADLS) is critical for enterprise security audits.

Private Link / Private Endpoints

By default, communications between planes occur over secure public cloud networks. FDEs implement **Private Link** (AWS) or **Private Endpoints** (Azure) to route traffic entirely over secure private cloud backbones, bypassing the public internet:

  • Front-End Connection: Restricts workspace UI access to corporate networks or VPNs.
  • Back-End Connection: Securely links the Databricks Control Plane to the Data Plane VPC without exposing public routing IPs.

Customer-Managed Keys (CMK)

Allows enterprises to encrypt cloud objects (DBFS storage bucket, notebook source code metadata) using their own KMS/Key Vault keys, ensuring data cannot be read by anyone else, including cloud provider administrators.

4. DBU Economics & Sizing Formulas

Databricks bills compute workloads based on **DBUs (Databricks Units)** consumed per hour, multiplied by the DBU cost of the workload type and the cloud provider VM charges.

📊 Pricing Comparison: All-Purpose vs. Jobs Compute

Workloads are billed differently depending on the cluster type. For AWS Premium/Enterprise tiers:
• All-Purpose Compute (Interactive Notebooks): ~$0.40 per DBU
• Jobs Compute (Automated Workflows): ~$0.15 per DBU (Almost 2.7x cheaper!)

Real-World DBU Sizing Math

Suppose a daily ingestion pipeline runs for 10 hours on a cluster of 4 nodes (plus 1 driver, totaling 5 VMs). Each VM size (e.g., m5.2xlarge) consumes 1.5 DBUs per hour. 

Total Cluster DBUs per Hour:
  5 VMs × 1.5 DBUs/hour = 7.5 DBUs/hour

Scenario A: Running as an Interactive Notebook (All-Purpose DBU rates)
  7.5 DBUs/hr × 10 hours × $0.40/DBU = $30.00 in DBU charges

Scenario B: Running as an Automated Workflow Job (Jobs DBU rates)
  7.5 DBUs/hr × 10 hours × $0.15/DBU = $11.25 in DBU charges

Weekly Savings (Moving to Jobs Compute):
  Interactive: $30.00 × 7 days = $210.00
  Automated: $11.25 × 7 days = $78.75
  FDE Cost Savings: $131.25 per week (62.5% Savings!)

This basic DBU math scales up to millions of dollars when applied across thousands of enterprise cluster hours.

5. Lab: Cluster Sizing & DBU Optimization

Objective

Design a cost-estimating spreadsheet for a nightly ingestion workflow and migrate it from interactive cluster execution to an automated job cluster.

Steps to Complete

  • Step 1 (Sizing): Analyze the average data volume of a nightly CSV upload (approx 50GB).
  • Step 2 (Selection): Identify an optimal node instance size. For typical memory-heavy joins, select memory-optimized VMs (e.g., r5.xlarge which consumes 1 DBU/hour).
  • Step 3 (Configuration): Open the Workflows UI in Databricks. Create a new task.
  • Step 4 (Migration): Link your PySpark notebook to this task. Rather than utilizing an active "All-Purpose" cluster, configure a brand new "Job Cluster" with 1 Driver and 3 Workers (total 4 nodes, 4 DBUs/hour).
  • Step 5 (Automation): Schedule the task to trigger at 2:00 AM daily.

Interview Questions & Model Answers

Explain the Control Plane vs Data Plane split and why it matters for security.
Testing: the #1 architecture fundamental.
The Control Plane is Databricks-managed (web UI, job scheduling, cluster management, metadata) and never processes customer data. The Data Plane is where compute runs and data is read/written — either Classic (VMs in the customer's own VPC/VNet, so data never leaves their network) or Serverless (Databricks-managed compute). This split lets enterprises keep data in their security perimeter while Databricks operates the platform.
Classic vs Serverless compute — how do you choose?
Testing: a real architecture decision.
Classic when strict compliance requires VMs inside the customer's own subnets or custom networking/instances. Serverless for fast startup, no idle cost, and lower ops — ideal for SQL warehouses and bursty jobs. Serverless folds infra cost into the DBU rate; Classic bills the cloud VMs separately.
Walk me through the full cost of a 10-hour, 5-node job as interactive vs a job cluster.
Testing: DBU math fluency.
Compute DBUs/hr = nodes × DBU/node. Multiply by hours and the per-DBU rate for the workload type. All-Purpose (interactive) has a much higher rate than Jobs compute, so moving the same cluster to a job cluster can cut the DBU charge ~2–3×, plus the cloud VM bill on top. I'd quantify both and recommend Jobs compute + autotermination.
What is Private Link and which connections does it secure?
Testing: enterprise networking.
Private Link (AWS) / Private Endpoints (Azure) route traffic over the cloud's private backbone instead of the public internet. Front-end secures user→workspace access (lock to corporate network); back-end secures Control Plane→Data Plane without public IPs. Combined with CMK, it satisfies strict security audits.
Why separate Dev/QA/Prod workspaces?
Testing: platform lifecycle.
Isolation of blast radius, data sensitivity, and permissions: developers explore freely in Dev, integration/regression runs in QA, and Prod is locked down to automated deploys only. I provision them identically via Terraform so config drift can't cause an audit failure or a "works in dev" surprise.
What can't you do on Community Edition, and how do you still demonstrate value?
Testing: tier honesty.
No custom VPC, Private Link, CMK, Unity Catalog, Workflows, or DBU billing. I'd say so plainly and demonstrate the concepts on a paid trial, or prove the compute/Spark fundamentals on Community and describe exactly how the enterprise features layer on.

Trial vs. Enterprise Limits

Databricks Community Edition is free but operates entirely in a shared control plane. You cannot connect cloud virtual networks, provision custom VPCs, configure Private Link, link Customer-Managed Keys, or access Workflows/DBU billing math.

To test network security, Private Link, and CMK, you must register for a Databricks 14-Day Full Premium/Enterprise Trial connected to a paid AWS, Azure, or Google Cloud account.
