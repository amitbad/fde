Structured Streaming — Incremental Processing Done Right

Streaming is where correctness bugs hide: duplicates, lost data, ever-growing state, and broken recovery. This chapter explains the micro-batch model, checkpoints, output modes, event-time windows and watermarks, Auto Loader for file ingestion, and how to achieve exactly-once semantics — the knowledge that separates an FDE from someone who copied a readStream snippet.

6.1 The Micro-Batch Mental Model

Structured Streaming treats an unbounded stream as a table that grows over time. Under the hood (default engine) it runs a series of small deterministic micro-batches: each trigger, it discovers new input, runs the same query as a batch, and commits progress.

source (files/Kafka) unbounded input table (conceptual) new data ─────► ┌───────────────────────────────┐ │ row row row ... (append only) │ └───────────────┬───────────────┘ every trigger: │ incremental query ┌──────────────▼───────────────┐ │ micro-batch N (a normal job) │ └──────────────┬───────────────┘ ▼ sink (Delta table) + checkpoint (offsets + state)

Because each micro-batch is an ordinary Spark job, everything from Chapter 2 (stages, shuffles, AQE) applies. The only new machinery is the checkpoint, which records what has been processed so the stream can recover exactly.

6.2 Checkpoints & Fault Tolerance

The checkpoint directory is the stream's durable memory. It stores source offsets (what input has been consumed) and state (for aggregations/joins). Combined with a replayable source and an idempotent sink, it delivers exactly-once end-to-end.

A minimal, correct streaming write
(spark.readStream
      .format("delta").table("bronze_events")
   .writeStream
      .format("delta")
      .option("checkpointLocation", "/chk/silver_events")   # MANDATORY, unique per stream
      .outputMode("append")
      .trigger(availableNow=True)                            # process all, then stop
      .toTable("silver_events"))

⚠️ The three cardinal checkpoint sins

  • Reusing one checkpoint for two streams → corrupted offsets, silent data loss.
  • Deleting the checkpoint to "reset" → the stream reprocesses everything from scratch (duplicates unless the sink dedupes).
  • Changing the query shape (adding a stateful op) against an existing checkpoint → incompatible state, stream won't start.

6.3 Output Modes & Triggers

Output mode Emits Valid with
append Only new rows that will never change. Non-aggregated streams, or aggregations bounded by a watermark.
update Only rows whose value changed this batch. Aggregations you want incrementally updated.
complete The entire result table every batch. Small aggregations only (state grows unbounded otherwise).
Trigger Behaviour Use
processingTime="5 minutes" Micro-batch every 5 min. Steady near-real-time.
availableNow=True Process all backlog, then stop. Incremental batch / cost-efficient scheduled jobs.
Continuous ~1 ms latency, limited ops. Rare, ultra-low-latency needs.

FDE cost tip

Trigger.AvailableNow on a scheduled job gives you streaming's exactly-once bookkeeping with batch economics — the cluster spins up, drains the backlog, and shuts down. This is the default recommendation unless a true latency SLA exists.

6.4 Event Time, Windows & Watermarks

Real events arrive late and out of order. To aggregate by event time without keeping state forever, you declare a watermark: "I will wait up to N for late data; anything later is dropped and its window state can be released."

Windowed aggregation with a watermark
from pyspark.sql.functions import window, col

(events
   .withWatermark("event_ts", "10 minutes")          # tolerate 10 min lateness
   .groupBy(window(col("event_ts"), "5 minutes"), col("device_id"))
   .count()
   .writeStream
   .outputMode("append")                              # window emitted once it closes past watermark
   .option("checkpointLocation", "/chk/win")
   .toTable("gold_device_counts"))
  • Without a watermark, stateful aggregation state grows forever → eventual OOM. The watermark is what lets Spark evict old state.
  • Tuning: a longer watermark captures more late data but holds more state (memory) and delays results. It is a correctness-vs-cost dial.

6.5 Auto Loader — Scalable File Ingestion

Auto Loader (cloudFiles) incrementally ingests new files from cloud storage without you tracking which files were seen. It scales to millions of files and handles schema inference and evolution.

Auto Loader with schema evolution
(spark.readStream.format("cloudFiles")
   .option("cloudFiles.format", "json")
   .option("cloudFiles.schemaLocation", "/schemas/orders")
   .option("cloudFiles.schemaEvolutionMode", "addNewColumns")
   .option("cloudFiles.inferColumnTypes", "true")
   .load("/Volumes/main/landing/orders/")
 .writeStream
   .option("checkpointLocation", "/chk/orders")
   .option("mergeSchema", "true")
   .trigger(availableNow=True)
   .toTable("bronze_orders"))
  • Directory listing mode (default): lists the folder — fine for modest file counts.
  • File notification mode: subscribes to cloud events (SNS/SQS, Event Grid) — required for very high file volumes; needs extra cloud permissions.
  • Rescued data column _rescued_data captures fields that didn't match the schema so nothing is silently lost.

6.6 Hands-On Lab — Recover a Stream Correctly

  1. Create a Delta table bronze_events and append a few rows.
  2. Start a stream from it into silver_events with a checkpoint and availableNow=True. Confirm row counts match.
  3. Append more rows to bronze and rerun the same stream. Confirm only the new rows are processed (offsets advanced) — not the whole table.
  4. Now delete the checkpoint directory and rerun. Observe every row reprocessed → duplicates in the sink. This demonstrates why checkpoints must never be casually deleted.
  5. (Cloud) Recreate the ingest with Auto Loader from a volume, drop new files in, and watch incremental pickup + the _rescued_data column on a malformed file.

Expected outcome

You will viscerally see that "the checkpoint is the stream's identity." This is the #1 streaming incident FDEs are called in to fix.

6.7 Interview Questions & Model Answers

How does Structured Streaming achieve exactly-once?
Testing: the central guarantee.
Three ingredients: a replayable source (offsets), a checkpoint that atomically records processed offsets and state, and an idempotent/transactional sink (Delta commits are atomic and keyed to batch id). On failure the stream replays from the last committed offset and the sink de-duplicates by batch id, so each record affects the result once.
What exactly is stored in a checkpoint?
Testing: recovery internals.
Source offsets (the progress log), commit log, and operator state for stateful queries (aggregations, joins, dedup). It's what allows recovery without reprocessing and why it must be unique per stream and preserved across restarts.
Explain watermarks. What breaks without one?
Testing: event-time correctness.
A watermark declares the max lateness tolerated for event-time aggregations, letting Spark finalize and evict old window state. Without it, stateful aggregation state grows unbounded and eventually OOMs, and append mode can't decide when a window is final.
append vs update vs complete output mode?
Testing: API correctness.
Append emits only finalized new rows (needs a watermark for aggregations). Update emits changed aggregate rows each batch. Complete re-emits the whole result table — only safe for small aggregates because state is unbounded.
Trigger.AvailableNow vs processingTime — why does it matter for cost?
Testing: FinOps thinking.
AvailableNow drains the backlog then stops, so a scheduled job gets exactly-once bookkeeping with batch economics (compute shuts down). processingTime keeps micro-batching on an interval, holding compute. I default to AvailableNow unless there's a genuine latency SLA.
What is Auto Loader and when do you use file notification mode?
Testing: ingestion at scale.
Auto Loader incrementally ingests new cloud files with schema inference/evolution and its own file-tracking. Directory listing is fine for modest volumes; file notification mode (cloud event queues) is needed at very high file counts to avoid expensive listings, at the cost of extra cloud IAM setup.
A customer deleted a checkpoint to "restart clean" and now has duplicates. What happened and how do you prevent it?
Testing: incident response.
Deleting the checkpoint erased the offset log, so the stream re-read the whole source and re-wrote rows. Prevention/fix: never delete checkpoints; if a reset is required, also truncate/rebuild the sink, or dedupe by a natural key + MERGE. Going forward, use idempotent sinks and document checkpoint locations as protected.
How do you handle a stream whose state keeps growing?
Testing: stateful ops hygiene.
Add/shorten a watermark so old state is evicted, ensure stream-stream joins have time constraints, use RocksDB state store for large state, and prefer append with bounded windows over complete mode.
Can you run Structured Streaming on Community Edition?
Testing: tier honesty.
Yes for core Delta-to-Delta streaming and rate sources — great for learning checkpoints and watermarks. Auto Loader and file-notification ingestion need a cloud-linked workspace.
