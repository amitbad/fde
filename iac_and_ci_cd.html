Workspace Automation (IaC) & automated CI/CD Pipelines

Deploy secure multi-workspace clusters and package end-to-end data workloads natively using Terraform and Databricks Asset Bundles (DABs).

Paid Cloud Subscription Required

1. Infrastructure as Code (IaC) via Terraform

Manual cluster provisioning via the web UI is a primary source of environment configuration drift and compliance audit failures. Forward Deployment Engineers declare the entire Databricks ecosystem using the official Databricks Terraform Provider.

Automating Core Resources

Terraform allows you to declare and version-control all enterprise platform components:

  • Workspace Provisioning: AWS VPCs, subnets, IAM Roles, and secure storage buckets.
  • User Directory Provisioning: Mapping AD/Okta directory groups to workspace access levels.
  • Standard Cluster Policies: Restricting standard developers from building massive, expensive virtual machines.
# Sample Terraform defining a secure, auto-terminating cluster
resource "databricks_cluster" "shared_etl" {
  cluster_name            = "Shared Production ETL"
  spark_version           = "14.3.x-scala2.12"
  node_type_id            = "m5.xlarge"
  autotermination_minutes = 15 # Shuts down idle nodes to save budgets

  autoscale {
    min_workers = 1
    max_workers = 8
  }
}

💡 Architecture Decision Pattern

Always include autotermination_minutes in your cluster declarations (recommended: 15-20 minutes). This forces idle clusters to spin down, saving clients thousands of dollars in accidental weekend VM runaways.

2. Databricks Asset Bundles (DABs)

While Terraform manages workspace infrastructure, **Databricks Asset Bundles (DABs)** are the modern way to package, test, and deploy code assets, dependencies, and execution tasks as a single declarative package.

The YAML Bundle Manifest

FDEs write a databricks.yml manifest detailing the environments (Dev, QA, Prod), task dependencies, cluster configurations, and artifacts. 

# Sample databricks.yml structure
bundle:
  name: billing-etl-pipeline

targets:
  dev:
    workspace:
      host: https://adb-dev.cloud.databricks.com
  prod:
    workspace:
      host: https://adb-prod.cloud.databricks.com

resources:
  jobs:
    nightly_billing_job:
      name: "Billing-ETL-Job"
      tasks:
        - task_key: run_transformation
          new_cluster:
            num_workers: 4
            node_type_id: r5.xlarge
          notebook_task:
            notebook_path: ./src/billing_transform.ipynb

FDE Value-Add: Decoupling Code from Workspace UI

Rather than manually importing and overwriting notebooks in the UI, developers work locally in IDEs (like VS Code), test with pytest, and deploy validated bundles directly using DABs command-line tools.

3. Automated Sprints & Git Integration

With code packaged in DABs and infrastructure in Terraform, FDEs build automated pipelines to test, validate, and deploy changes on commit.

Typical GitHub Actions Workflow

  1. Developer Commit: A developer pushes code changes to a feature/ branch.
  2. Continuous Validation: GitHub Actions triggers, checks code style with luff, and validates the bundle manifest structure.
  3. Automated Testing: Runs local unit tests using mocks (mocking PySpark dataframes).
  4. Approval & Auto-Deploy: Merging to the main branch automatically triggers DABs to deploy the package directly to the target environment's workflow tasks.
# Sample GitHub Actions Step deploying with Databricks CLI
- name: Deploy Production Bundle
  run: databricks bundle deploy -t prod
  env:
    DATABRICKS_HOST: ${{ secrets.PROD_HOST }}
    DATABRICKS_TOKEN: ${{ secrets.PROD_TOKEN }}

4. Lab: Building and Deploying a DABs Package

Objective

Initialize a local Databricks Asset Bundle, validate its structural configuration, and deploy it to your test workspace.

Steps to Complete

  • Step 1 (CLI Setup): Download and install the Databricks CLI. Configure a local profile linking your workspace token:
    databricks configure --token
  • Step 2 (Init): Initialize a brand new bundle directory structure using the built-in template:
    databricks bundle init default-python
  • Step 3 (Refactor): Examine the created databricks.yml and place your source transformation python files into the /src directory.
  • Step 4 (Validation): Verify your YAML syntax is completely valid using the CLI:
    databricks bundle validate
  • Step 5 (Deployment): Deploy the bundle assets as a test job directly to your development target:
    databricks bundle deploy -t dev

Interview Questions & Model Answers

Terraform vs Databricks Asset Bundles — what does each own?
Testing: the core tooling split.
Terraform provisions long-lived infrastructure — workspaces, metastores, cluster policies, users/groups, networking. DABs package and deploy code assets — jobs, DLT pipelines, notebooks, wheels — across dev/QA/prod targets. Terraform builds the house; DABs move the furniture in repeatably.
Why declare cluster policies in Terraform instead of the UI?
Testing: IaC value.
Version control, review, and identical enforcement across environments — no config drift or audit surprises. Policies also enforce guardrails (max nodes, forced autotermination, allowed instance types, mandatory cost tags), so developers can't accidentally create runaway or untagged clusters.
Walk me through a CI/CD pipeline for a Databricks project.
Testing: end-to-end delivery.
Push to a feature branch triggers GitHub Actions: lint, databricks bundle validate, and unit tests (pytest on extracted logic). On merge to main, the pipeline runs databricks bundle deploy -t prod with secrets injected, deploying jobs/pipelines. Prod is deploy-only; humans don't edit it directly.
How do you unit test Spark code that reads production tables?
Testing: software-engineering rigor.
Extract transformations into pure functions that take and return DataFrames (no hardcoded reads). Test them with a local SparkSession on small in-memory fixtures, asserting on output rows/schema. I/O is isolated at the edges and mocked, so tests are fast and deterministic in CI.
How do you manage secrets across environments?
Testing: security hygiene.
Never in code. Use Databricks secret scopes (optionally backed by a cloud key vault) referenced at runtime, and CI secrets (GitHub Actions encrypted secrets) for deploy tokens. Terraform variables for infra secrets come from a secure backend, never committed state.
Why package transformations as a Python wheel instead of importing notebooks?
Testing: production code structure.
A wheel is versioned, testable, dependency-managed, and reusable across jobs — unlike notebook %run imports with hidden global state. It enables real unit testing, semantic versioning, and clean rollback, which is what production delivery requires.

Trial vs. Enterprise Limits

Databricks Community Edition does not support workspace APIs, cluster policies, multi-environment deployments, or Databricks CLI integration, making DABs and Terraform testing impossible.

To write and deploy Terraform scripts or use DABs bundles, you must register for a Free Trial subscription on a paid cloud provider to access full workspace admin API rights.
